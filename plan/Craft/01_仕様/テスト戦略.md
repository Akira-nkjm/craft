# テスト戦略

> 親: [[最終構成]] / 関連: [[対処方針]] / [[UnifiedRegistry設計]] / [[API設計]]

各層のテストの位置付け、特に **E2E が何を保証するか** を明確化。

---

## テストピラミッド

```
       ┌───────────┐
       │    E2E    │  少数（5〜15 シナリオ）、遅い、価値高
       ├───────────┤
       │Integration│  中程度、API + registry + TOML
       ├───────────┤
       │ Contract  │  Pydantic snapshot
       │   Unit    │  多数、速い、純粋関数
       └───────────┘
```

---

## 各層の役割

### Unit
- 対象: `fld()` / `_strictify()` / `writer.atomic_write` / `dynamic_enum` 等の純粋関数
- データ: テスト関数内に直書きの小さい dict / TOML 文字列
- 例: 「`atomic_write` が tmp file → rename している」「`load_instance_enum` が TOML キーから enum を作る」

### Contract (snapshot)
- 対象: Component base class が生成する Pydantic モデル
- 何を見るか: `Battery.Entry.model_json_schema()` の JSON を **snapshot 比較**
- 価値: 「Component / trait をいじったら生成 schema が壊れた」を即検出
- ツール: `syrupy` / `pytest-snapshot`

### Integration
- 対象: 複数モジュールを組み合わせた挙動
- 例: 「FastAPI ルーター + UnifiedRegistry + atomic writer」「POST → TOML 書き込み → ETag 返却」
- データ: `tmp_path` に書き出す **テスト専用 TOML**

### Property-based
- 対象: 不変条件（add → remove で TOML が元に戻る、書き込み順序が結果に影響しない等）
- ツール: `hypothesis`

### **E2E (本ノートの主題)**
- 対象: **実際のユーザシナリオ全体**を、外部から叩いて結果を検証

## E2E が想定するシナリオ

E2E は **API/CLI/MCP を「ユーザと同じインターフェース」で叩く** ことが本質。
本プロジェクトでは **3 系統 × 各 3〜5 シナリオ** を想定。

### 系統 A: CLI からの典型ワークフロー

**A-1: ハッピーパス（新規 component 追加 → 検証）**
```bash
# 1. 起点: クリーンな TOML
# 2. CLI でバッテリー追加
craft create power battery aux --data '{"spec":{...},"design":{...}}'
# 3. 一覧で見える
craft get power battery
# 4. verify を走らせて全 pass
craft verify power
# 5. 期待: exit 0、stdout に "PASS" を含む
```

**A-2: バリデーションエラー（負の容量を弾く）**
```bash
craft create power battery bad --data '{"spec":{"capacity_wh":-10}}'
# 期待: exit != 0、stderr に "validation_error" / "capacity_wh" / "greater_than_equal"
```

**A-3: 同名作成で 409 conflict**
```bash
craft create power battery main --data '...'   # 既存
# 期待: exit != 0、"already exists" 系メッセージ
```

**A-4: 削除 → 検証で要求未達**
```bash
craft delete power battery main
craft verify power
# 期待: exit != 0、未達要求リスト出力
```

**A-5: analysis 実行**
```bash
craft analysis run power total_power_by_mode --mode safe
# 期待: 数値出力、 exit 0
```

### 系統 B: HTTP API（curl 相当）

FastAPI の `TestClient` で十分。実サーバ起動は不要。

**B-1: CRUD ラウンドトリップ**
- POST 作成 → GET 取得 → PATCH 更新 → GET で更新確認 → DELETE → GET で 404

**B-2: ETag による楽観的ロック**
- GET で ETag 取得 → PATCH (If-Match) で成功 → 古い ETag で PATCH → 412

**B-3: schema endpoint が JSON Schema を返す**
- `GET /api/schema/power/battery` → `model_json_schema()` と一致

**B-4: verify 結果が runs に保存される**
- POST `/api/verify/power` → job_id → ポーリング → result.toml 生成確認

**B-5: introspection endpoint**
- `GET /api/introspect/components-with-field?path=design.power_modes` → 期待するリスト

### 系統 C: MCP プロトコル

`mcp` Python SDK の `Client` でローカル MCP サーバを叩く。

**C-1: tools/list で全コンポーネント tool が見える**
- 期待: `add_battery`, `add_solar_panel` 等が列挙

**C-2: tool 呼び出しで TOML が書き換わる**
- `add_battery({...})` → 実 TOML 反映 → 戻り値に id / etag

**C-3: validation error が tool error で返る**
- 不正引数 → tool error response、stderr に validation 情報

**C-4: resources/list で現在の TOML 内容が見える**
- 期待: `resource://components/power/batteries/main` が読める

**C-5: analysis を tool として実行できる**
- `analyse_power_budget({mode:"safe"})` → 数値結果

## E2E で使う TOML

### 「現実 TOML を使うか fixture か」の方針

| データ源 | 用途 |
|---|---|
| **`tests/snapshots/minimal/`** | 1 サブシステムだけの最小構成、ほぼ全 E2E が使う |
| **`tests/snapshots/realistic/`** | 実プロジェクトに近い規模、月 1 で同期更新 |
| **本物の `design/data/`** | 使わない（CI 環境差・データ変更で壊れる） |

→ ユーザ要望通り **「テストは別 TOML」**。`tests/snapshots/` 配下に snapshot 化したものだけ使う。

### snapshot の更新フロー

1. 本物 TOML に重要な構造変更があった
2. 開発者が `make sync-test-snapshots` を実行
3. `tests/snapshots/realistic/` が更新される
4. テストが落ちたら設計変更の影響として人間が判断

---

## 何を E2E でテスト「しない」か

- パフォーマンス（負荷試験は別フレームワーク）
- veriq 内部ロジック（veriq 側のテストに任せる）
- 個別 Pydantic validator（unit でカバー）
- UI（Phase 2 で playwright を入れたら）

---

## CI での E2E

- 系統 A (CLI) と B (HTTP) は **毎 PR で実行**
- 系統 C (MCP) は **mainブランチでのみ実行**（MCP サーバ起動コストあり）
- 全 E2E トータルで **2分以内** を目標

---

## まとめ

- **E2E は「ユーザになりきって API/CLI/MCP を叩く」**シナリオ群
- ハッピーパス + 主要エラー経路 + 重要な業務フロー (verify, analysis)
- データは `tests/snapshots/` の専用 TOML、本物データは使わない
- 3 系統 (CLI / HTTP / MCP) で **計 10〜15 シナリオ** が当面のスコープ
