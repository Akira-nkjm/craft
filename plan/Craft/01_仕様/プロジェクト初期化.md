# プロジェクト初期化 — `craft init` CLI

> 親: [[最終構成]] / 関連: [[データパイプライン]] / [[コンポーネントデコレータ仕様]] / [[Config設計]] / [[実装テンプレート/README]]

新規プロジェクトの立ち上げと既存プロジェクトへの system 追加を **1 コマンド** で済ませる CLI。
ユーザが手書きするボイラープレートを最小化し、概念設計の初手から思考に集中できるようにする。

---

## 1. 動機

「[[最終構成]] を読んで Component を書き始めるまで」の障壁を取り除く。現状の手作業（[[実装テンプレート/README]] 参照）:

1. `systems/<name>/` ディレクトリ作成
2. `components.py` / `configs.py` / `analyses.py` / `scope.py` / `__init__.py` を空ファイルから書き起こす
3. 親 `systems/__init__.py` に `project.add_scope(...)` を追加
4. `data.toml` 用に空ファイル作成

→ 5〜6 ファイル × system 数。初心者の **「どこから書き始めるか」のつまづき** にも繋がる。

---

## 2. コマンド一覧

```bash
craft init project [--template TEMPLATE] [--path DIR]
craft init system <name> [--template TEMPLATE]
craft init component <system> <name> [--multi] [--traits TRAIT,...]
craft init config <system> <name>
craft init analysis <system> <name> [--verify]
```

### 2.1 `init project`

新規プロジェクト全体を生成。

```bash
craft init project                       # default = empty
craft init project --template cubesat
craft init project --template microsat --path ./my_satellite
```

| Template | 含まれる system | 想定用途 |
|---|---|---|
| `empty` | （なし） | 自分で全部作る |
| `cubesat` | `power`, `cdh`, `com`, `payload`, `mission` | 1U〜3U CubeSat |
| `microsat` | 上記 + `thermal`, `aocs`, `propulsion`, `structure` | 100kg 級小型衛星 |

生成されるルート構造:

```
my_satellite/
├── pyproject.toml              # craft を依存に持つ
├── README.md                   # プロジェクト固有 README
├── .gitignore                  # generated/, __pycache__ 等
├── systems/
│   ├── __init__.py             # project = vq.Project(...) + add_scope を自動生成
│   └── <system>/...         # template に応じて
└── (generated/ は実行時に作成)
```

### 2.2 `init system <name>`

既存プロジェクトに system を 1 つ追加。

```bash
craft init system aocs
craft init system propulsion --template hardware    # hardware-only
craft init system mission --template config-only    # Config のみ
```

| Template | 生成ファイル | 用途 |
|---|---|---|
| `default` | `components.py`, `analyses.py`, `scope.py`, `__init__.py`, `data.toml` | 通常 hardware system |
| `hardware` | 上記から `configs.py` を省く | hardware 専用 |
| `config-only` | `configs.py`, `scope.py`, `__init__.py`, `data.toml` | Config のみ（mission, environment 等） |

**重要な副作用**: 親 `systems/__init__.py` に **自動追記**:

```python
# systems/__init__.py（追記後）
import veriq as vq

from .power.scope import power
from .aocs.scope import aocs           # ← 自動追加された行

project = vq.Project("MySatellite")
project.add_scope(power)
project.add_scope(aocs)                # ← 自動追加された行
```

→ 追記は **idempotent**（同じ system を再 init しても重複しない）。

### 2.3 `init component <system> <name>`

既存 system の `components.py` に **1 class を追記**。

```bash
craft init component power battery --multi --traits TemperatureSensitive,PowerConsuming
craft init component cdh obc                                       # default = Singleton
```

追記される雛形:

```python
# systems/power/components.py（追記）

class Battery(Component, MultiInstance, TemperatureSensitive, PowerConsuming):
    """TODO: Battery の説明を書く。"""

    # TODO: Spec field を定義
    # 例: capacity_wh: float = fld(ge=0, unit="Wh")

    class Design:
        # TODO: Design field を定義
        pass

    class Requirements:
        # TODO: Requirements field を定義（不要なら削除）
        pass
```

→ class を追加した後、`craft scaffold --component battery` で対応する `data.toml` セクションを生成（[[データパイプライン]] §4）。

### 2.4 `init config <system> <name>` / `init analysis <system> <name>`

`configs.py` / `analyses.py` に同様の追記。

```bash
craft init config mission mission_profile
craft init analysis power total_power_by_mode --verify   # @analysis(verify=True)
```

## 3. テンプレートの中身

### 3.1 `empty` プロジェクト

最小ファイルセット。system は無く、ユーザが `init system` で足していく。

```
my_satellite/
├── pyproject.toml
├── README.md
├── .gitignore
└── systems/
    └── __init__.py             # project = vq.Project("MySatellite") のみ、scope 登録なし
```

### 3.2 `cubesat` テンプレート

CubeSat の典型 system セットを **空クラスで** 生成。class 名だけ用意、field はユーザが埋める。

```
systems/
├── __init__.py                 # 5 つの scope が登録済み
├── power/
│   ├── __init__.py
│   ├── components.py           # class Battery, SolarPanel, PDM (空 body)
│   ├── analyses.py             # 空 (@analysis 待ち)
│   ├── scope.py                # build_system_root_model("power") 呼び出し
│   └── data.toml               # 空（craft scaffold で雛形を埋める）
├── cdh/
│   └── components.py           # class OBC (Singleton, 空 body)
├── com/
│   └── components.py           # class Transceiver, Antenna
├── payload/
│   └── components.py           # class Camera (placeholder)
└── mission/
    ├── configs.py              # class MissionProfile, OrbitalParameters
    └── ...
```

### 3.3 `microsat` テンプレート

`cubesat` + `thermal` / `aocs` / `propulsion` / `structure`。各々の典型 component を空クラスで配置。

### 3.4 テンプレートのカスタマイズ

テンプレート定義は `craft/templates/<name>/` 配下の YAML / ディレクトリツリーで管理:

```
craft/templates/cubesat/
├── manifest.yaml               # system 一覧、各 system の component 一覧
└── files/                      # 実ファイル雛形（jinja2 で変数展開）
    ├── pyproject.toml.j2
    ├── README.md.j2
    └── systems/...
```

→ ユーザ独自テンプレートは `~/.craft/templates/<name>/` または `./templates/<name>/` に置けば `--template <name>` で参照可能（Phase 2）。

## 4. 生成ファイルの中身（詳細）

### 4.1 `systems/__init__.py`（project 統合）

```python
"""Subsystem registration for the craft project."""

import veriq as vq

# system scope の import（init system 実行ごとに 1 行追加される）
from .power.scope import power
from .cdh.scope import cdh
# from .<new_subsystem>.scope import <new_subsystem>   ← init 時にここに追加

project = vq.Project("MySatellite")

project.add_scope(power)
project.add_scope(cdh)
# project.add_scope(<new_subsystem>)                   ← init 時にここに追加
```

### 4.2 `systems/<sub>/scope.py`

```python
"""<Sub> system veriq scope."""

# 注意: from __future__ import annotations は書かない

import veriq as vq

from craft.schema import build_system_root_model
from craft.paths import system_data_path


<sub> = vq.Scope("<sub>")


@<sub>.root_model()
class <Sub>RootModel(build_system_root_model("<sub>", system_data_path("<sub>"))):
    """Auto-built from registry; do not add fields here."""
    pass
```

→ `<sub>` 部分はテンプレート展開で system 名に置換。新 component を追加しても本ファイルは触らない（[[実装テンプレート/systems/power/scope]] と同じ）。

### 4.3 `systems/<sub>/components.py`

```python
"""<Sub> system components."""

from craft.schema import Component, MultiInstance, fld
from craft.schema.traits import (
    PowerConsuming,
    TemperatureSensitive,
)


# template 由来の空 class（フィールドはユーザが埋める）
# 削除可、または `craft init component` で追加可
```

### 4.4 `systems/<sub>/data.toml`

```toml
# === systems/<sub>/data.toml ===
# Auto-managed by `craft scaffold`.
# 値は API/CLI 経由での編集を推奨。直接編集も可。

# (空。`craft scaffold` で雛形が入る)
```

### 4.5 `pyproject.toml`

```toml
[project]
name = "my_satellite"
version = "0.1.0"
requires-python = ">=3.11"
dependencies = [
    "craft>=0.1.0",
    "veriq>=0.4.2",
]

[tool.craft]
project_name = "MySatellite"
```

```

.venv/

# craft generated artifacts
generated/merged.toml
generated/runs/

# keep merged.lock (CI gate)
!generated/merged.lock
```

## 5. 既存プロジェクトでの動作

### 5.1 既存ファイルの保護

- `init project` で既にディレクトリが存在する場合 → **422 エラー**（`--force` でも上書きせず、`init system` を使うよう案内）
- `init system` で既に同名 system が存在する場合 → **422 エラー**
- `init component <sub> <name>` で同名 class が既に `components.py` に存在 → **422 エラー**（ユーザが手で消すか別名にすることを案内）

→ 破壊的操作は CLI から「絶対に」発火しない。

### 5.2 idempotent 追記

`systems/__init__.py` への追記は AST ベース:

1. ファイルを Python AST でパース
2. `from .<sub>.scope import <sub>` import 文が既にあるか確認
3. `project.add_scope(<sub>)` 呼び出しが既にあるか確認
4. 無いものだけ追加して、ファイルを書き戻し（元のコメント・順序を可能な限り保持）

→ ユーザが手で並べ替えても再 init で壊れない。

## 6. init 後のフロー

```
craft init project --template cubesat
   ↓
craft scaffold                       # 各 data.toml に field 雛形を埋める
   ↓
[エディタで data.toml の値を埋める or craft patch で API 経由編集]
   ↓
craft verify                         # merge + veriq 検証
   ↓
craft runs latest                    # 結果確認
```

→ **初回 4 コマンドで「動く設計」に到達**。

## 7. API（オプション、Phase 2 検討）

CLI ファーストで、API は当面検討対象外。理由:
- ファイルシステム操作なので API 経由よりも CLI が自然
- 既存プロジェクトに対して system を加える操作は権限管理が必要

ただし将来的に MCP tool として LLM から `init_subsystem` を呼べるようにする可能性はある（[[MCP設計]] §残る論点）。

## 8. 確定事項

| 項目 | 決定 |
|---|---|
| サブコマンド | `init project / system / component / config / analysis` の 5 種類 |
| Project テンプレート | `empty` / `cubesat` / `microsat` の 3 種類を built-in |
| テンプレート保管場所 | `craft/templates/<name>/` (manifest.yaml + jinja2) |
| Subsystem テンプレート | `default` / `hardware` / `config-only` の 3 種類 |
| 既存ファイル保護 | 既存と衝突したら **422 エラー、--force でも上書きしない** |
| `systems/__init__.py` 追記 | AST ベースで idempotent |
| `data.toml` への field 自動投入 | しない（init は class 定義のみ、`craft scaffold` の責務） |
| API | Phase 1 では提供しない（CLI のみ） |

## 9. 残る論点

- **ユーザカスタムテンプレート** — `~/.craft/templates/` をどの段階で対応するか（Phase 2 候補）
- **大学/研究室固有テンプレート** — 燈研究室や宇宙系研究室のテンプレートを built-in に含めるか、別パッケージにするか
- **AST 追記の限界** — ユーザが大幅にフォーマットを変更した `systems/__init__.py` への追記が失敗するケース
  - 暫定: 失敗時は diff を表示してユーザに手動マージを促す
- **`init component` の trait セット指定** — `--traits PowerConsuming,TemperatureSensitive` は文字列、typo に気付けない
  - 案: tab 補完を整備、または `craft registry list-traits` で確認可
- **既存 component を rename** — `craft rename component` のような migration CLI も将来必要
