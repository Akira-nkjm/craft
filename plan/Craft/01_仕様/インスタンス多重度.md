# インスタンス多重度 — Singleton (default) と MultiInstance の扱い

> 親: [[最終構成]] / 関連: [[コンポーネントデコレータ仕様]] / [[Config設計]] / [[宣言とTOMLの対応表]] / [[実装テンプレート/_internals/trait一覧]]

`class X(Component):` の **default は Singleton**。複数積む場合は `MultiInstance` trait を明示する。

---

## 1. 方針サマリ

| 宣言 | TOML 構造 | 用途 |
|---|---|---|
| `class OBC(Component):` | `[obc.spec] / [obc.design]` フラット | 1 個しか積まない hardware（OBC, BusStructure, …） |
| `class Battery(Component, MultiInstance):` | `[batteries.spec]` 共有 + `[batteries.<name>.design]` 個別 | 同型を複数積む hardware（Battery, SolarPanel, …） |
| `class Camera(Component, MultiInstance, shared_spec=False):` | `[cameras.<name>.spec] / [cameras.<name>.design]` 全て個別 | 異種を複数積む hardware（型違いの 2 カメラ等） |

→ **「物理的に 1 個か複数か」を Python 宣言だけで決める**。default が Singleton なので、概念設計の最初の一歩は迷わない。

---

## 2. なぜ Singleton を default にするか

### 2.1 概念設計の自然な順序

衛星の概念設計は **「OBC を 1 つ、Battery を 1 つ、Antenna を 1 つ、…」** とまず単数で組み上げ、後から「これは冗長化したい」「これは複数積みたい」と数を増やしていくのが自然。

→ default が Multi-instance だと、最初から `batteries.main` のような placeholder 名を強要され、`.main` が冗長な dead weight になる。

### 2.2 「`.main` placeholder 問題」の解消

旧 default (Multi) で頻発していた:

```toml
[obc.main.spec]           # OBC は普通 1 個なのに .main が冗長
[bus_structure.main.spec] # BusStructure も同上
```

→ default Singleton なら `[obc.spec]` で済む。

### 2.3 Multi-instance の意図を明示化

「これは複数積む」が `MultiInstance` trait として **コードに現れる** ので、レビュー時に冗長系・複数化の判断が見える。

---

## 3. 候補案の比較（採用判断の根拠）

### 案 A: 旧 default = Multi-instance、`Singleton` trait で opt-in（旧採用案）
- `.main` placeholder が概念設計初期から強要されるのが直観に反する
- → ❌ 廃案

### 案 B: 新 default = Singleton、`MultiInstance` trait で opt-in **✅ 採用**
- 概念設計の自然な順序と一致
- 「数を増やす」判断がコードに明示される
- TOML は default で意図通りフラット

### 案 C: 別概念 `Config` を導入
- Config は Singleton 限定で **既に存在**（[[Config設計]]）。non-hardware の Singleton を表す
- Component (hardware) の多重度問題は本ノートで扱う

### 案 D: 動的にどちらにも対応（registry が "instance 数 1 なら flat、2+ なら nested"）
- TOML 構造が状態依存になり予測不能 → ❌

## 4. `MultiInstance` trait の挙動

### 4.1 宣言

```python
# Multi-instance、同型製品で spec 共有（default）
class Battery(Component, MultiInstance, TemperatureSensitive):
    capacity_wh: float = fld(ge=0, unit="Wh")
    nominal_voltage_v: float = fld(ge=0, unit="V")

    class Design:
        depth_of_discharge: float = fld(ge=0, le=1)
```

```python
# Multi-instance、instance ごとに spec も異なる場合
class Camera(Component, MultiInstance, shared_spec=False):
    sensor_size_mm: float = fld(ge=0)
    pixel_count: int = fld(ge=0)

    class Design:
        mounting_angle_deg: float = fld()
```

### 4.2 自動変化

| 観点 | default (Singleton) | `MultiInstance` trait あり |
|---|---|---|
| TOML の root key | singular (`obc`, `battery_pack`) | plural (`batteries`, `cameras`) |
| TOML 構造 | `[obc.spec] / [obc.design]` | `[batteries.spec]` 共有 + `[batteries.<name>.design]` (or 全個別) |
| インスタンスキー | **なし** | あり |
| Registry の cardinality | `"singleton"` | `"multi"` |
| API path | `/components/cdh/obc` | `/components/power/battery/{name}` |
| 追加可能数 | 1（既存があれば 409） | 任意 |
| 削除 | "リセット" 相当 | 個別削除 |
| ETag | component 単位 | インスタンス単位 |

### 4.3 TOML キー名の上書き

不規則 plural（`antenna` → `antennae` 等）や、Singleton で class 名と TOML キーを別にしたい場合:

```python
# Multi-instance: plural= で上書き
class SBandAntenna(Component, MultiInstance, plural="sband_antennas"):
    ...

# Singleton: key= で上書き
class OBC(Component, key="onboard_computer"):
    ...
```

→ `key=` (Singleton) / `plural=` (MultiInstance) の使い分けで意味的に正確。

## 5. インスタンス名規約（MultiInstance のみ）

### 5.1 意味のある名前を使う

| 良い例 | 悪い例 |
|---|---|
| `batteries.primary`, `batteries.secondary` | `batteries.main`, `batteries.aux` |
| `solar_panels.port`, `solar_panels.starboard` | `solar_panels.panel_a`, `panel_b` |
| `heaters.battery_pack`, `heaters.fuel_line` | `heaters.heater_1`, `heater_2` |

→ **placeholder 名 (`main` / `1` / `2`) は「これ default の Singleton で良いのでは?」のサイン**。

### 5.2 機械的列挙

完全同等で個別差異がない（4 枚同形 solar panel 等）:

```toml
[solar_panels.spec]              # 共有 spec
efficiency = 0.28

[solar_panels.string_1.design]
placement = "port_upper"
[solar_panels.string_2.design]
placement = "port_lower"
[solar_panels.string_3.design]
placement = "starboard_upper"
[solar_panels.string_4.design]
placement = "starboard_lower"
```

→ 規約: prefix + 通し番号、または部位名（`port_upper`, `starboard_lower`）。

### 5.3 推奨 vs 強制

- 規約: **推奨**（lint で警告を出す程度）
- 強制: しない（試作・初期段階の placeholder 命名は許容）

## 6. `MultiInstance` / `Singleton` の実装イメージ

```python
# schema/traits.py

class MultiInstance:
    """インスタンスが複数存在しうる component。

    TOML 構造に instance キーが入り (`[batteries.<name>...]`)、
    plural 形のテーブル名が使われる。
    """

    __cardinality__ = "multi"


# Singleton は trait ではなく default。明示が要らない。
# 旧 Singleton trait は廃止（default のため不要）。
```

`Component.__init_subclass__` 内で:

```python
cardinality = "multi" if MultiInstance in cls.__mro__ else "singleton"

defn = ComponentDefinition(..., cardinality=cardinality)

# TOML キーの決定
if cardinality == "singleton":
    toml_key = key_override or cls.__name__.lower()       # "obc"
else:
    toml_key = plural_override or _auto_pluralize(cls.__name__)  # "batteries"

defn.toml_key = toml_key
```

→ writer / reader / router が `cardinality` を見て分岐する。core の 1 箇所で吸収。

## 7. API への影響

### 7.1 Endpoint 差分

| 操作 | Singleton (default) | MultiInstance |
|---|---|---|
| 一覧 | （該当無し、または `GET /components/cdh/obc` で 1 個返す） | `GET /components/power/battery` |
| 取得 | `GET /components/cdh/obc` | `GET /components/power/battery/{name}` |
| 作成 | `POST /components/cdh/obc`（既存なら 409） | `POST /components/power/battery/{name}` |
| 更新 | `PUT /components/cdh/obc` | `PUT /components/power/battery/{name}` |
| 削除 | `DELETE /components/cdh/obc`（リセット相当） | `DELETE /components/power/battery/{name}` |

### 7.2 vq.Ref への影響

```python
# Singleton (default)
obc: Annotated[OBC.Entry, vq.Ref("$.obc")]

# MultiInstance
spec: Annotated[Battery.Entry, vq.Ref("$.batteries.primary")]
batteries: Annotated[vq.Table[BatteriesName, Battery.Entry], vq.Ref("$.batteries")]
```

→ Ref の path に instance キー部分が入るかどうかの違いだけ。構文は同じ。

## 9. 例: どう使い分けるか

### default (Singleton) になる Component (hardware)

「物理的に 1 個しか積まない」と判断したら **trait なし** で書く:

- `OBC` — On-Board Computer
- `BusStructure` — バス構造本体
- `PropulsionModule` — 推進系本体（タンク含む 1 アセンブリ）
- `PayloadDataHandler` — ペイロード処理機（1 機の場合）
- `ThermalController` — 熱制御コントローラ

冗長化したくなったら `MultiInstance` を後付けして migration。

### `MultiInstance` を付ける Component (hardware)

「複数積む」が確定または高確率の hardware:

- `Battery` — 主/副、冗長系（同型 → shared_spec=True default）
- `SolarPanel` — 複数パドル / 複数 string（同型）
- `Heater` — 配置複数
- `Antenna` — LGA / HGA / 複数指向
- `Transceiver` — 主/副 冗長
- `Camera` / `Payload` — 通常複数（異種なら `shared_spec=False`）
- `ReactionWheel` — 通常 3〜4 基
- `StarTracker` — 2〜3 基

### Config（別概念）

Component ではなく `Config` を使う:

- `MissionProfile`, `OrbitalParameters`, `AttitudeControlLaw`, `RadiationEnvironment`, `MassBudgetTargets`

→ [[Config設計]]。Config は常に Singleton（multi 化なし）。

### 迷ったら → default (Singleton)

「複数積むかわからない」段階では **trait なし** で書き始める。
複数化が確定した時点で `MultiInstance` を後付け（migration script で TOML 変換）。

## 10. 確定事項

| 項目 | 決定 |
|---|---|
| 多重度の default | **Singleton**（trait なしの `class X(Component):`） |
| Multi-instance の opt-in | **`MultiInstance` trait**（旧 `Singleton` trait は廃止、default のため不要） |
| Singleton の TOML 構造 | **フラット**（`[obc.spec] / [obc.design]`） |
| MultiInstance の TOML 構造 | 既存通り（`[batteries.<name>.spec]` or `[batteries.spec]` 共有 + `[batteries.<name>.design]`） |
| インスタンス名規約 | 意味のある名前を推奨、placeholder（`main` 等）は **Singleton 化のサイン** |
| TOML キー指定引数 | `key=` (Singleton) / `plural=` (MultiInstance) |
| 同 instance 重複作成 | Singleton は **409 Conflict** |
| 削除 | Singleton は「リセット」相当（型は残る、値が空） |

---

## 11. 残る論点

- **`Singleton` 削除の意味論** — 「リセット」と「完全削除」の差は API 上どう表現するか
  - 案: `DELETE` は値クリア、component 定義削除は不可
- **`Singleton` → `MultiInstance` への移行** — 「途中で冗長化したくなった」時の手順
  - 案: migration script を書く、TOML 構造が変わるので破壊変更
- **migration ツール** — `[obc.spec]` を `[obcs.primary.spec]` に展開する逆方向 (Singleton → Multi) と、Multi → Singleton（1 個に縮退）の双方が必要か
- **既存の `class X(Component, Singleton):` 表記** — 旧 Singleton trait は default 化したため不要だが、書かれていてもエラーにせず警告留め（移行猶予）
