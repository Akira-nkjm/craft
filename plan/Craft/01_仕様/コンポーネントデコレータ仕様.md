> 📌 **採用方式 = 方式 C（Component base class、decorator 廃止）**
> ファイル名に「デコレータ」が残っているのは旧リンク維持のため。

## 1. 基本形

`Component` の default は **Singleton**（1 機構成）。複数積む場合は `MultiInstance` trait を明示する。

```python
from craft.schema import Component, MultiInstance, fld
from craft.schema.traits import TemperatureSensitive

# (a) Singleton (default): OBC など 1 機構成
class OBC(Component):
    """On-Board Computer (1 機)"""

    clock_mhz: int = fld(ge=0, unit="MHz")

    class Design:
        firmware_version: str = fld(default="")

# (b) MultiInstance: Battery など複数積む
class Battery(Component, MultiInstance, TemperatureSensitive):
    """二次電池（複数積み、同型製品共有 spec）"""

    capacity_wh: float = fld(ge=0, unit="Wh", desc="Battery capacity")
    nominal_voltage_v: float = fld(ge=0, unit="V", default=0.0)

    class Design:
        depth_of_discharge: float = fld(ge=0, le=1)

    class Requirements:
        depth_of_discharge_max: float = fld(default=0.8, gt=0, le=1)
```

- `Component` を継承するだけで registry 登録・Pydantic モデル生成が走る（`__init_subclass__`）
- **default cardinality は Singleton** — `MultiInstance` trait を付けない場合は 1 機構成（[[インスタンス多重度]]）
- `subsystem` は **ファイルパスから自動推論**（`subsystems/power/components.py` → `"power"`）
- 明示する場合: `class Battery(Component, MultiInstance, subsystem="power"):`

## 2. trait（多重継承で機能合成）

| trait | 効果 | spec に増える field |
|---|---|---|
| `MultiInstance` | 複数インスタンスを許可（plural テーブル化） | — |
| `TemperatureSensitive` | 動作温度範囲を持つ | `operating_temp_min_c`, `operating_temp_max_c` |
| `PowerConsuming` | 消費電力を持つ | `default_power_consumption_per_unit_w` |
| `HasPowerMode` | `OperationMode` ごとの design 値 | `Design.<field>_by_mode: dict[OperationMode, float]` |
| `SpecOnly` | Design 自動生成をスキップ（material library 等） | — |

> **default cardinality は Singleton**（trait 不要、TOML が `[obc.spec]` フラット）。
> 「同型製品を複数積み spec を共有」は `MultiInstance` trait + default の `shared_spec=True` で実現（[[宣言とTOMLの対応表]] §P3）。
> 旧 `Singleton` trait は廃止（default のため不要）、旧 `WithDatasheet` trait 案も採用見送り。

詳細は [[実装テンプレート/_internals/trait一覧]] と [[インスタンス多重度]] / [[宣言とTOMLの対応表]] を参照。

旧 `@component(mixin="temperature", has_power_mode=True, base="power_spec")` は廃止：
- `mixin="temperature"` → `TemperatureSensitive` 継承
- `has_power_mode=True` → `HasPowerMode` 継承
- `base="power_spec"` → `PowerConsuming` 継承

→ **文字列引数を型に乗せる**（typo は ImportError）。これが方式Cの本質的利点。

## 3. plural 上書き

TOML テーブル名は通常 `Battery` → `batteries` のように自動英語複数化される。
不規則名は class 引数で上書き：

```python
class SBandAntenna(Component, plural="sband_antennas"):
    ...
```

## 4. 生成物

| 名前 | 内容 |
|------|------|
| `<Name>.Spec` | spec field の Pydantic モデル（class 本体 + trait の spec field） |
| `<Name>.Design` | inner `Design` クラスから生成 |
| `<Name>.Requirements` | inner `Requirements` クラスから生成（任意） |
| `<Name>Entry` | `ComponentEntry[Spec, Design]` + `requirements` フィールド |
| registry エントリ | `UnifiedRegistry.components[("<subsystem>", "<name>")]`（[[UnifiedRegistry設計]]） |
| **Subsystem Model への寄与** | 当該 subsystem の root_model に `<plural>: vq.Table[..., <Name>.Entry]` field が自動追加される（MultiInstance）または `<name>: <Name>.Entry` field（Singleton）。詳細は [[最終構成]] §2「Subsystem Model」 |

→ `Battery.spec` / `Battery.design` 属性経由でアクセスする設計。
→ `BatterySpec` を独立 import する必要はない（型 stub の対象外）。
→ **設計者は `<Name>Model` を手書きしない**。veriq の root_model は `build_subsystem_root_model(subsystem)` が registry から動的に組み立てる（[[実装テンプレート/subsystems/power/scope]]）。

## 5. `fld()` ヘルパー（統一）

```python
fld(
    *,
    ge=None, le=None, gt=None, lt=None,
    default=NOT_SET,
    unit: str | None = None,
    desc: str | None = None,
    group: str | None = None,        # UI hint
    order: int | None = None,        # UI hint
) -> FieldInfo
```

- 旧 `cf()` / `af()` / `qf()` は廃止 → **`fld()` 一本に統一**（[[最終構成]] §5.1）
- `unit` は description に `[Wh]` 形式で埋め込み → JSON Schema 経由でフロントに伝達
- `pydantic.Field(...)` を返す

## 6. 一律適用される挙動

- **`extra="forbid"`**: `_strictify()` で Spec / Design / Requirements / Entry / RootModel すべてに適用
- Web 入力・TOML 直書きの両方で typo を `ValidationError` で早期検出
- 例外は `meta` フィールド配下のみ（§7）

---

## 7. `meta` フィールド（自由メモ領域）

全 Component に `meta: dict[str, Any] | None = None` を **自動付与**。
`extra="forbid"` を維持しつつユーザの自由メモを許容する逃げ道。

```toml
[batteries.main]
capacity_wh = 100
manufacturer = "Panasonic"

[batteries.main.meta]                # ← 構造は何でも OK
notes = "選定理由: 過去実績"
datasheet_url = "https://..."
reviewed_by = "akira"
reviewed_date = "2026-05-22"
```

- `meta` 配下は registry の対象外（schema 検証なし）
- API レスポンスにも素通しで含める
- `meta` 配下の typo は当然検出されない（仕様）
- 規約: 単純な scalar / dict のみ推奨。複雑構造を積み始めたら正式フィールド化を検討

## 9. 設計判断の根拠 — なぜ base class か（旧「型検証と宣言方式」要約）

> 詳細な比較・経緯は archive [[型検証と宣言方式]] を参照。本節はその要約。

### 9.1 pyrefly に何が「見える」か

| 観点 | 引数あり decorator (旧) | base class + trait (現役 = 方式 C) |
|---|---|---|
| decorator/継承 シグネチャ | ✅ `@overload` で表現可 | ✅ 単純 |
| **機能スイッチの効果** | ❌ 文字列引数のため型情報に乗らない (`mixin="temperature"` の typo は実行時エラー) | ✅ `TemperatureSensitive` 継承で typo は ImportError |
| trait の field（例: 動作温度範囲） | ❌ 見えない | ✅ MRO に乗る、hover/go-to-definition 可 |
| 生成される `BatterySpec` / `BatteryEntry` | ❌ 動的、stub 必須 | ⚠️ stub 必須は同じ（ただし `Battery.spec` 属性経由で多くは不要） |

### 9.2 「機能スイッチを型に乗せる」が本質的利点

旧 `@component(mixin="temperature", has_power_mode=True, base="power_spec")` の文字列引数は **型チェッカに伝わらない**。
これを **trait の多重継承** (`class Battery(Component, TemperatureSensitive, HasPowerMode, PowerConsuming):`) に置き換えると:

- 文字列 typo が **ImportError** になる
- trait が定義する field（例: `operating_temp_min_c`）が **MRO 経由で見える**
- IDE のコンストラクタ補完が効く

### 9.3 `dataclass_transform` (PEP 681) との相性

`Component` を `@dataclass_transform` でマークすると、pyrefly / pyright が field 宣言の型を正しく扱う（Pydantic v2 / SQLAlchemy 2.0 / msgspec が採用済みの実績ある手法）。

### 9.4 analysis は decorator のまま

`@analysis` は関数なので継承が使えない。decorator のまま残す（引数は `verify` / `cache` 等の制御フラグのみ、機能スイッチ的な文字列引数は持たない）。
詳細は [[Analysis詳細仕様]]。

### 9.5 from __future__ import annotations 禁止

`from __future__ import annotations` を入れると **annotation が文字列化** され、veriq の `inspect.signature` が型を解決できなくなる → 禁止。
ruff + grep スクリプトで lint 強制（[[対処方針]] §B.2）。
