# データパイプライン — TOML merge と scaffold

> 親: [[最終構成]] / 関連: [[コンポーネントデコレータ仕様]] / [[Config設計]] / [[UnifiedRegistry設計]] / [[API設計]] / [[veriq仕様メモ]]

> 📌 **階層関係**: Subsystem (= veriq Scope) → **Subsystem Model** (= veriq root_model、registry から自動派生) → Component / Config (個々の宣言) → TOML data。
> Model 層は [[最終構成]] §2「Subsystem Model」参照。本ノートは Model 層と data 層の橋渡し（merge / scaffold）を扱う。

ユーザが触る `subsystems/<name>/data.toml` 群と、veriq が消費する **単一の入力 TOML** との橋渡し。
2 つの機能を扱う:

1. **merge** — 全サブシステムの `data.toml` を `generated/merged.toml` に統合
2. **scaffold** — Pydantic モデル（Component / Config の定義）から `data.toml` の雛形を生成・整形

---

## 1. 動機

### 1.1 merge が必要な理由

[[veriq仕様メモ]] の `vq.evaluate_project()` は **単一 TOML** を入力に取る。一方、本プロジェクトのユーザは subsystem ごとに分かれた `subsystems/<name>/data.toml` を編集する（[[最終構成]] §4）。両者を繋ぐ層が必要。

```
subsystems/power/data.toml    ┐
subsystems/thermal/data.toml  ┼──► merge ──► generated/merged.toml ──► vq.evaluate_project()
subsystems/com/data.toml      │
subsystems/cdh/data.toml      │
subsystems/mission/configs    ┘   (Config も含む)
```

### 1.2 scaffold が必要な理由

新 Component / Config を Python で宣言した直後、対応する `data.toml` セクションを **手書きするのは間違いやすい**:

- フィールド名の typo（`capacity_wh` を `capacityWh` と書く等）
- trait が自動付与する field の漏れ（`PowerConsuming` の `default_power_consumption_per_unit_w` を忘れる）
- ネスト構造の組み立てミス（`[batteries.spec]` vs `[batteries.main.design]` の階層）

→ **registry から TOML 雛形を自動生成** すれば、ユーザは値を埋めるだけで済む。既存 TOML がある場合は **差分追加 + 整形** する。

## 2. `generated/` ディレクトリ

```
craft/
├── subsystems/                  # ユーザが編集する元データ
│   ├── power/data.toml
│   ├── thermal/data.toml
│   └── ...
│
├── generated/                   # ★ merge / 派生物の出力先（git 管理は方針による）
│   ├── merged.toml             # 全 subsystem を統合した単一 TOML（veriq 入力）
│   ├── merged.lock             # 元 data.toml の sha256 を記録（再生成判定用）
│   └── runs/                   # veriq の出力ディレクトリ
│       ├── <run_id>/
│       │   ├── result.toml     # vq.export_to_toml の出力 (input + .calc + .verification 統合)
│       │   ├── input.toml      # この run の入力スナップショット (merged.toml のコピー)
│       │   ├── meta.json       # run metadata (status / duration / errors / registry sha)
│       │   └── trace.json      # build_traceability_report の出力
│       ├── latest -> <run_id>/ # 最新 run への symlink
│       └── analyses/           # @analysis(cache=True) のキャッシュ
│           └── <name>/<inputs_hash>.toml
│
└── ...
```

### 2.1 git 管理ポリシー

- **`merged.toml` は git 管理しない**（`.gitignore`）。derived artifact なので元ファイルから常に再生成可能
- `merged.lock` は管理する（CI で「元 TOML から再生成しても変わらない」を gate）
- `generated/runs/` は管理しない（[[対処方針]] §D.1 と整合、必要時のみ別管理）

### 2.1.1 run_id の形式

```
<ISO8601 timestamp (秒精度、コロンはハイフン)>-<short hash>
例: 2026-05-26T15-42-18Z-a3f2c8
```

- timestamp は UTC、ファイルシステムに優しい `T15-42-18Z` 形式
- short hash は `(input sha256 + registry sha256)[:6]` — 同一入力で 2 度 run しても hash 衝突は許容（timestamp で区別）
- 並び替えで時系列順になるよう先頭は timestamp（lexicographic = chronological）

### 2.1.2 run の中身

| ファイル | 内容 |
|---|---|
| `result.toml` | veriq の `export_to_toml` 出力。`[<scope>.model.*]` + `[<scope>.calc.*]` + `[<scope>.verification.*]` を 1 ファイルに統合 |
| `input.toml` | この run 時点の merged.toml のスナップショット。後から「何を入力に走らせたか」を辿るため |
| `meta.json` | `{"status": "success", "duration_s": 1.42, "started_at": "...", "errors": [], "registry_sha": "..."}` |
| `trace.json` | `vq.build_traceability_report()` の JSON 化（要求 ↔ 検証のマッピング） |

### 2.1.3 latest symlink

- 最新 run 完了時に `generated/runs/latest` を更新（atomic rename）
- Windows でも `os.symlink` が使えない場合は `latest.txt` に run_id を書く fallback
- `@analysis` の `vq.Ref("@calcname")` 解決時に latest を参照

### 2.1.4 analyses キャッシュ

`@analysis(cache=True)` の結果は **run と独立した** `generated/runs/analyses/<name>/<inputs_hash>.toml` に保存:

- run 内で完結する verification とは別経路。全 run 横断で共有
- キャッシュキー = `name + code_version + canonicalize(direct_inputs) + refs_hash`（[[Analysis詳細仕様]] §4）
- 古いキャッシュの掃除は CLI（`craft runs prune-cache --older-than 30d`）

### 2.2 出力フォーマット — veriq の `[<Scope>.model.<...>]` 形式

veriq は TOML 入力を **`[<ScopeName>.model.<root_model_field>...]`** で受け取る（[[veriq仕様メモ]] §TOML 入出力の構造）。
`merged.toml` もこの規約に揃える:

```toml
# === from subsystems/power/data.toml ===
[power.model.batteries.spec]                    # MultiInstance + shared_spec=True
capacity_wh = 100.0
[power.model.batteries.main.design]
depth_of_discharge = 0.7
[power.model.batteries.main.design.power_modes]
safe = true
nominal = true
[power.model.batteries.aux.design]
depth_of_discharge = 0.6

# === from subsystems/thermal/data.toml ===
[thermal.model.heaters.battery_pack.spec]
rated_power_w = 5.0

# === from subsystems/cdh/data.toml ===
[cdh.model.obc.spec]                            # Singleton はインスタンスキー無し
clock_mhz = 100
[cdh.model.obc.design]
firmware_version = "v1.2.0"

# === from subsystems/mission/data.toml ===
[mission.model.mission_profile]                 # Config
duration_years = 5
[mission.model.orbital_parameters]
semi_major_axis_km = 6978
```

- セクション名衝突（同名 component が同 subsystem 内に複数）は元 TOML パースで弾く
- subsystem (= scope) 単位で独立した名前空間なので、`power.model.battery` と `cdh.model.battery` のような同名共存は OK（実用上はやらない）
- 元ファイル由来のコメントは保持する努力をする（toml ライブラリの能力に依存）
- 出力先頭に generation timestamp + 元ファイル sha256 一覧を comment で書く

#### 出力側（veriq の計算結果）

veriq は `vq.export_to_toml` で **同じ TOML に `.calc.` / `.verification.` セクションを追記** する:

```toml
# 入力（merged.toml）
[power.model.batteries.spec]
capacity_wh = 100.0

# 出力（result.toml = merged.toml + 以下）
[power.calc.calculate_power_margin]
margin = 10.0

[power.verification.verify_battery_capacity]
passed = true
```

→ 入力と出力で **TOML パスの 2 段目（`.model.` / `.calc.` / `.verification.`）を切り替える** ことで衝突しない設計。

### 2.3 Subsystem Model（root_model）との対応

veriq は **各 Scope (= subsystem) に root_model を要求**（[[最終構成]] §2、[[veriq仕様メモ]]）。
`merged.toml` の各セクションが **どの subsystem の root_model のどの field か** は TOML パスから直接読める:

```
merged.toml のパス                                  │ 所属 │ root_model field
────────────────────────────────────────────────────┼──────┼──────────────────────────
[power.model.batteries.*]                            │ power │ PowerModel.batteries: vq.Table[...]
[power.model.solar_panels.*]                         │ power │ PowerModel.solar_panels: vq.Table[...]
[power.model.pdms.*]                                 │ power │ PowerModel.pdms: vq.Table[...]
[thermal.model.heaters.*]                            │ thermal │ ThermalModel.heaters: vq.Table[...]
[cdh.model.obc.*]                                    │ cdh   │ CdhModel.obc: OBC.Entry (Singleton)
[mission.model.mission_profile]                      │ mission │ MissionModel.mission_profile: Config
[mission.model.orbital_parameters]                   │ mission │ MissionModel.orbital_parameters: Config
```

→ veriq の `vq.load_model_data_from_toml(project, merged.toml)` は **top-level の scope 名で切り分け → `.model.` 配下を root_model に流し込む**。
→ 各 subsystem の data.toml も同じ `[<subsystem>.model.<...>]` で書く（scaffold が生成する形式もこれに統一）。

## 3. merge 機能

### 3.1 CLI

```bash
craft merge                           # 全 subsystem を generated/merged.toml に出力
craft merge --check                   # diff があれば exit 1（CI 用）
craft merge --output path/to/out.toml # 出力先指定
craft merge --subsystem power thermal # 部分 merge
craft merge --dry-run                 # stdout に出力、ファイル書き込みなし
```

### 3.2 API

| Method | Path | 説明 |
|---|---|---|
| POST | `/api/projects/{pid}/merge` | merge 実行、ETag 付き |
| GET | `/api/projects/{pid}/merged` | 最新の merged.toml を返す |
| GET | `/api/projects/{pid}/merged?subsystem=power` | サブシステム部分のみ抽出 |

### 3.3 動作仕様

```
1. UnifiedRegistry を走査 → 全 subsystem の (toml_key, cardinality, plural/key) を収集
2. 各 subsystems/<name>/data.toml を読み込み（無ければ空テーブル）
3. キー衝突チェック:
   - 同一 toml_key が複数 subsystem に存在 → MergeConflict
   - 同一 subsystem 内の重複は read 時の TOML パースエラーで弾く
4. 順序を安定化（subsystem 名アルファベット順、内部は registry 登録順）
5. 元ファイル sha256 を merged.lock に記録
6. generated/merged.toml に atomic write
```

### 3.4 自動再生成のタイミング

- `craft verify` 実行時に自動 merge（差分があれば）
- API の `POST /verify` も同じ
- 明示的に merge したい場合は CLI / API
- CI gate: `craft merge --check` で merged.lock の整合性を確認

### 3.5 エラー設計

| エラー | 原因 | 解決 |
|---|---|---|
| `MergeConflict` | 同一 `toml_key` が複数 subsystem | Component / Config の名前を変える、または `plural=` / `key=` で明示 |
| `MissingTomlFile` | `data.toml` が無い subsystem を merge 対象に含めた | scaffold を先に実行、または `--subsystem` で除外 |
| `TomlParseError` | 元 TOML の構文エラー | 元ファイルを修正 |
| `SchemaValidation` | TOML が registry の schema と不一致 | スキーマと合わせる、または scaffold で再整形 |

## 4. scaffold 機能

### 4.1 CLI

```bash
# 雛形生成（既存に無い項目を追加）
craft scaffold                          # 全 subsystem
craft scaffold --subsystem power        # 指定 subsystem のみ
craft scaffold --component battery      # 指定 component のみ（subsystem 推論）

# 整形のみ（field の順序・コメント・空行を統一）
craft scaffold --format-only

# 既存値はそのまま、欠落 field だけ追加
craft scaffold --add-missing            # default の動作

# 値の上書き禁止モード（既存 field の値を絶対に触らない）
craft scaffold --preserve-values        # default ON

# 強制再生成（既存値も default で上書き、危険）
craft scaffold --overwrite              # 確認プロンプト付き
```

### 4.2 API

| Method | Path | 説明 |
|---|---|---|
| POST | `/api/projects/{pid}/scaffold` | scaffold 実行（add-missing + format） |
| POST | `/api/projects/{pid}/scaffold/{subsystem}` | サブシステム単位 |
| GET | `/api/projects/{pid}/scaffold/preview/{subsystem}` | 書き込まずに結果を返す |

### 4.3 動作仕様

```
1. UnifiedRegistry から対象 subsystem の (Component, Config) 一覧を取得
2. 各 component / config について:
   a. registry が想定する TOML テーブル名・field 名・型・default 値を構築
   b. 既存 data.toml を読む（無ければ空）
   c. diff を計算:
      - 既存にあるが registry に無い field → 警告（削除はしない、ユーザ判断）
      - registry にあるが既存に無い field → 追加（default 値、無い場合はコメント `# TODO: required` + placeholder）
      - 両方にあり値が registry の制約 (ge/le/enum) を満たす → 既存値保持
      - 両方にあり値が制約違反 → 警告 + 既存値保持（破壊しない）
   d. field 順序を registry 順に整形
   e. コメントを registry の `desc` から自動付与（既存コメントが優先）
3. atomic write
```

### 4.4 雛形の例

新規 component を宣言した直後の scaffold 結果:

```python
# subsystems/power/components.py
class Battery(Component, MultiInstance, PowerConsuming, TemperatureSensitive):
    capacity_wh: float = fld(ge=0, unit="Wh", desc="Battery capacity")
    nominal_voltage_v: float = fld(ge=0, unit="V", default=3.7)

    class Design:
        depth_of_discharge: float = fld(ge=0, le=1)

    class Requirements:
        depth_of_discharge_max: float = fld(default=0.8, gt=0, le=1)
```

```toml
# subsystems/power/data.toml （scaffold 直後、veriq の [scope.model.<...>] 形式）

# Battery — 複数積まれる component（subsystem=power = veriq Scope "power"）
[power.model.batteries.spec]
capacity_wh = 0.0                                                # TODO: required, ge=0 [Wh]
nominal_voltage_v = 3.7                                          # [V]
default_power_consumption_per_unit_w = 0.0                       # TODO: required, ge=0 [W] (PowerConsuming)
operating_temperature_min_c = 0.0                                # TODO: required (TemperatureSensitive)
operating_temperature_max_c = 0.0                                # TODO: required (TemperatureSensitive)

# Instance: 最低 1 つ書く必要あり。コピーして個別 design を入れる。
[power.model.batteries.main.design]
depth_of_discharge = 0.0                                         # TODO: required, ge=0 le=1

[power.model.batteries.main.design.power_modes]                  # PowerConsuming 自動付与
safe = false
nominal = false
science = false

[power.model.batteries.main.requirements]
depth_of_discharge_max = 0.8                                     # gt=0 le=1
```

### 4.5 既存ファイル更新時の挙動

```toml
# 既存 data.toml（capacity_wh だけ書かれている）
[power.model.batteries.spec]
capacity_wh = 120.0
```

↓ Component に `manufacturer: str = fld(default="")` を追加して scaffold 実行

```toml
[power.model.batteries.spec]
capacity_wh = 120.0                # ← 既存値保持
manufacturer = ""                  # ← 新規追加（default 値）
default_power_consumption_per_unit_w = 0.0  # ← もともと無かった必須 field も追加
# ...
```

→ **既存値は絶対に変更しない**（`--overwrite` 明示時のみ上書き）。

### 4.6 整形ルール

| 項目 | 規約 |
|---|---|
| field 順序 | registry の宣言順（class 本体 → trait 由来 → Design → Requirements → meta） |
| インデント | TOML 標準（テーブル直下の field は indent なし） |
| コメント | `# <desc> [unit]`（field 直後に付ける、既存コメント優先） |
| `# TODO:` マーカー | 必須 field で default 値が無いものに自動付与 |
| 空行 | テーブル間に 1 行、コンポーネント間に 2 行 |
| 文字列クオート | double quote 統一 |
| 数値表記 | `1.0` 形式（`1` ではなく）。float は明示的に小数点 |

## 5. merge と scaffold の関係

```
[Component / Config 宣言を追加・変更]
            │
            ▼
   craft scaffold              ← 各 subsystem の data.toml を更新
            │
            ▼
   [ユーザが TODO を埋め、値を編集]
            │
            ▼
   craft merge                  ← generated/merged.toml を出力
            │
            ▼
   craft verify (= vq.evaluate_project(merged.toml))
            │
            ▼
   generated/runs/<id>/result.toml
```

scaffold と merge は **独立した責務**:
- scaffold = registry → data.toml （sources of truth: Python 宣言）
- merge = data.toml × N → merged.toml （sources of truth: 各 data.toml）

両者を強制的に連結しない（scaffold せずに手書きで data.toml を作っても merge は動く）。

## 6. 実装構造

```
craft/
├── core/
│   ├── merge.py                # MergeEngine 本体
│   ├── scaffold.py             # ScaffoldEngine 本体
│   ├── toml_io.py              # 既存（atomic write / lock）
│   └── toml_formatter.py       # 共通整形ロジック（merge / scaffold 両方が使う）
├── cli/
│   ├── merge.py                # `craft merge ...`
│   └── scaffold.py             # `craft scaffold ...`
└── api/routers/
    ├── merge.py                # /api/projects/{pid}/merge
    └── scaffold.py             # /api/projects/{pid}/scaffold
```

### 6.1 共通インターフェース

```python
@dataclass
class MergeResult:
    output_path: Path
    source_files: dict[Path, str]      # path → sha256
    conflicts: list[MergeConflict]
    written: bool                      # --check 時は False
    diff: str | None                   # 既存との diff（CI / preview 用）


@dataclass
class ScaffoldResult:
    subsystem: str
    files_updated: list[Path]
    added_fields: dict[str, list[str]]   # toml_path → added field names
    removed_warnings: list[str]          # registry に無い既存 field
    constraint_violations: list[str]     # 既存値が制約違反
    diff: str
```

→ いずれも JSON 化可能、API レスポンスとして返せる。

### 6.2 TOML ライブラリ

- パースは `tomllib` (Python 3.11+ 標準) — 読み取り専用、コメント保持しない
- 書き込みは `tomlkit`（コメント保持、format 制御可能）
- **コメント保持が必要な場面が多いので統一して `tomlkit` を使う** ことを検討

## 8. 確定事項

| 項目 | 決定 |
|---|---|
| 出力先 | `generated/` ディレクトリ |
| merge 出力 | `generated/merged.toml`（git 管理外）+ `merged.lock`（管理） |

| 共通 formatter | `core/toml_formatter.py` に集約、merge / scaffold 両方が使う |

## 9. 残る論点

- **scaffold 時の instance 命名** — Multi-instance component の初期 instance キー（`main` を入れるか、コメントで誘導するか）
  - 暫定: コメント `# instance 名を決めてください` + `<INSTANCE_NAME>` placeholder
- **既存値の制約違反検出** — scaffold で `ge=0` 違反を検出した時、修正提案まで出すか
- **複数プロジェクト対応** — Phase 2 で `/projects/{pid}/...` 化した時の `generated/` 位置
  - 暫定: `data/projects/{pid}/generated/`
- **コメント保持の限界** — `tomlkit` でも複雑な編集後にコメントが消えるケースあり。受け入れる線引きを実装着手時に決定
- **merge と scaffold の双方向性** — `craft scaffold` で `data.toml` を作った後、`craft merge` で逆引き（`merged.toml` を編集 → 各 `data.toml` に反映）したいか
  - 暫定: 一方向のみ（merged.toml は read-only、編集は data.toml 経由）
