# 型検証と宣言方式 — pyrefly フレンドリーな書き方

> ⚠️ **archive**: 結論と核心は [[コンポーネントデコレータ仕様]] §9「設計判断の根拠」に統合済み。
> 本ノートは方式A/B/C の全比較と設計経緯の詳細記録として残置。

> 関連: [[コンポーネントデコレータ仕様]] / [[実装テンプレート/README]]
> 旧提案（経緯）: [[構成見直し案]]

「decorator に引数を取ると pyrefly が見れなくなるのでは?」という問いへの精密回答。
**結論: 引数自体は問題なし。問題は decorator がクラスを「変換」すること**。
さらに **decorator を使わない方式（`class Battery(Component, ...):`）の方が型に厳しくできる**。

---

## 1. 何が型チェッカに「見える」か

pyrefly / pyright / mypy が `Battery` を解析する時、見えるのは:

1. ✅ `class Battery:` の本体（field 宣言）
2. ✅ 継承している base class（MRO 全体）
3. ✅ decorator のシグネチャ（戻り値型）
4. ❌ **decorator の中身が実行時に何をするか**
5. ❌ decorator が動的に生成したクラス（`BatterySpec` 等）

→ **基底クラスに書いたこと**は見える、**decorator の挙動**は見えない。
これが最重要のメンタルモデル。

---

## 2. 3 つの方式の比較

### 方式 A: 引数あり decorator

```python
@component(system="power", base="power_spec", mixin="temperature")
class Battery:
    capacity_wh: float = fld(ge=0, unit="Wh")
```

#### pyrefly に見えるもの
- `Battery` クラスの field 宣言
- `component` のシグネチャ (`@overload` で表現可能)

#### 見えないもの
- `system="power"` が結果に与える影響
- `base="power_spec"` で `default_power_consumption_per_unit_w` が増えるかどうか
- `mixin="temperature"` で温度フィールドが増えるかどうか
- 生成された `BatterySpec` / `BatteryEntry`（gen_stubs が `.pyi` を吐かない限り）

#### 評価
- ✅ 引数を取ること自体は問題なし（`@overload` で書ける）
- ❌ **文字列引数の効果は型チェッカに伝わらない** → typo しても気付かない
- ❌ 機能スイッチが型情報にならない

---

### 方式 B: 引数なし decorator + trait 多重継承（[[構成見直し案]] 採用案）

```python
@component
class Battery(TemperatureSensitive):
    capacity_wh: float = fld(ge=0, unit="Wh")
```

#### pyrefly に見えるもの
- `Battery` クラスの field 宣言
- **`TemperatureSensitive` の継承** → MRO に乗る、属性も型で見える
- `component` のシグネチャ（引数なしなので単純）

#### 見えないもの
- 生成された `BatterySpec` / `BatteryEntry`（gen_stubs 必須は同じ）

#### 評価
- ✅ trait の **typo 不可能**（import エラー）
- ✅ trait の **意味が型システムに乗る**（hover, go-to-definition が効く）
- ✅ decorator は引数 0 個で `@overload` すら不要
- ✅ pyrefly に **完全に見える機能スイッチ**

---

### 方式 C: decorator 廃止、base class だけ（Pydantic / SQLAlchemy 流）

```python
from craft.schema import Component, TemperatureSensitive, fld


class Battery(Component, TemperatureSensitive):
    """二次電池"""

    capacity_wh: float = fld(ge=0, unit="Wh")
    nominal_voltage_v: float = fld(ge=0, unit="V", default=0.0)

    class Design:
        depth_of_discharge: float = fld(ge=0, le=1)

    class Requirements:
        depth_of_discharge_max: float = fld(default=0.8, gt=0, le=1)
```

`Component` の `__init_subclass__` で registry 登録 + Pydantic モデル生成。
**decorator は無い**。

#### pyrefly に見えるもの
- `Battery` のすべて
- `Component` / `TemperatureSensitive` の継承
- `Component` が `dataclass_transform` でマークされていれば、 **生成された field の型まで自動的に正しく扱われる**
- IDE で `Battery(capacity_wh=100, ...)` のコンストラクタ補完も効く

#### 見えないもの
- 内部生成される独立クラス `BatterySpec` 単体（ただし `Battery.spec`, `Battery.design` 属性経由でアクセスする設計にすれば不要）

#### 評価
- ✅ **最も型チェッカに友好的**
- ✅ Pydantic / SQLAlchemy 2.0 / msgspec 等が **この方式を採用している実績**
- ✅ `dataclass_transform` (PEP 681) で完全な型サポート
- ✅ decorator が無い分、認知負荷も低い
- ⚠️ `Component` という base class を毎回書く必要（が、トレードオフは僅か）

## 3. `dataclass_transform` の威力

PEP 681 で導入された **「このクラスは dataclass-like な field 宣言を持つ」** と型チェッカに伝える機構。

```python
from typing import dataclass_transform

@dataclass_transform(field_specifiers=(fld,))
class Component:
    """全 component の基底。__init_subclass__ で registry 登録"""

    def __init_subclass__(cls, **kwargs):
        # ここで Pydantic モデル生成、registry 登録
        ...
```

これにより pyrefly が:
- `Battery(capacity_wh=100, nominal_voltage_v=3.7)` のシグネチャを自動推論
- `fld()` が pydantic.Field 相当だと理解
- field の型・default・制約を全部追跡

→ Pydantic v2 がこれで `BaseModel` のコンストラクタを正しく型推論できる根拠と同じ。

## 4. system 自動推論との両立

方式 C でも `system` 自動推論は可能:

```python
class Component:
    def __init_subclass__(cls, *, system: str | None = None, **kwargs):
        if system is None:
            system = _infer_from_caller_file()
        ...
```

または **明示時はキーワード引数**:

```python
class Battery(Component, TemperatureSensitive, system="power"):
    ...
```

PEP 487 の `__init_subclass__` キーワード引数。明示派でも 1 行ですっきり。

## 5. trait の自動効果（方式 B / C 共通）

trait は **マーカーであり、フィールド宣言**:

```python
class TemperatureSensitive:
    """動作温度範囲を持つことを示す"""
    operating_temperature_min_c: float = fld(unit="degC")
    operating_temperature_max_c: float = fld(unit="degC")


class PowerConsuming:
    default_power_consumption_per_unit_w: float = fld(ge=0, unit="W")


class HasPowerMode:
    # Design に power_modes が増える
    # （実装は Component の __init_subclass__ が trait を見て Design を再構築）
```

→ trait に書かれた field は **MRO 経由で Battery にマージ**される。
→ pyrefly が `Battery.operating_temperature_min_c` を **本物の属性として認識**。

## 6. 比較表

| 観点 | 方式 A (decorator+args) | 方式 B (decorator+trait) | 方式 C (subclass only) |
|---|---|---|---|
| 文字列引数の typo 検出 | ❌ 実行時のみ | ✅ import 時 | ✅ import 時 |
| 機能スイッチが型に乗る | ❌ | ✅ | ✅ |
| trait field が型に乗る | ❌ | ✅ | ✅ |
| 生成 model が型に乗る | gen_stubs 必須 | gen_stubs 必須 | dataclass_transform で **不要** |
| `Battery(...)` 補完 | ❌（生成モデル別） | ❌（同） | ✅ |
| 学習コスト | 中（引数多い） | 低 | 低（Pydantic 経験者は最低） |
| 業界標準パターン | 古典的 | やや独自 | ✅ Pydantic/SQLAlchemy 流 |
| decorator 必要 | あり | あり（引数なし） | なし |

→ **方式 C が一番強い**。

## 7. 推奨

### ✅ 方式 C 採用

[[構成見直し案]] の方式 B からさらに進めて、 **decorator を廃止して `Component` base class** に統一する。

```python
# systems/power/components.py

from craft.schema import Component, fld
from craft.schema.traits import (
    PowerConsuming,
    TemperatureSensitive,
    HasPowerMode,
)


class Battery(Component, TemperatureSensitive):
    capacity_wh: float = fld(ge=0, unit="Wh")
    nominal_voltage_v: float = fld(ge=0, unit="V", default=0.0)

    class Design:
        depth_of_discharge: float = fld(ge=0, le=1)

    class Requirements:
        depth_of_discharge_max: float = fld(default=0.8, gt=0, le=1)


class SolarPanel(Component, PowerConsuming, TemperatureSensitive):
    area_m2: float = fld(ge=0, unit="m^2")
    efficiency: float = fld(ge=0, le=1, default=0.28)

    class Design:
        cell_count: int = fld(ge=1)


class PDM(Component, PowerConsuming, HasPowerMode):
    rated_current_a: float = fld(ge=0, unit="A")

    class Design:
        efficiency: float = fld(ge=0, le=1, default=0.95)
```

### `@analysis` はどうするか

`@analysis` は **関数** に貼るので、class base trick が使えない。
ただし関数 decorator は class decorator より型に優しい（戻り値の型を `@overload` で表現すれば、関数本体は変わらないので IDE 補完も効く）。

→ **`@analysis` は decorator のまま**。ただし [[構成見直し案]] のとおり `system` 自動推論を入れる。

```python
@analysis(imports=["orbital"])    # system 自動、verify/cache は flag
def total_power_by_mode(...) -> float: ...
```

## 8. 影響範囲（[[構成見直し案]] からの差分）

| 項目 | 構成見直し案（旧） | 本ノート（新） |
|---|---|---|
| Component の宣言 | `@component` decorator + 多重継承 | `class X(Component, ...)` base のみ |
| 引数なし | OK | OK（さらに不要） |
| trait | trait 多重継承 | 同じ |
| `system` 自動推論 | `inspect.stack()` | `__init_subclass__` 内で `inspect.stack()` |
| Analysis | `@analysis` decorator | **変わらず** decorator |

→ Component のみ「decorator を完全廃止」。Analysis は decorator 継続。

## 11. 残る論点

- **`Component` の `__init_subclass__` 実装複雑度** — Pydantic v2 をどう内部で使うか
- **trait の field 衝突解決** — `TemperatureSensitive` と `PowerConsuming` が同名 field を持ったらどうするか
- **Component の MRO** — `Component` を **必ず先頭に置く規約**（Python の MRO 仕様上、左から解決される）
- **Pydantic v2 との直接統合** — `Component` を `BaseModel` 直接派生にするか、内部で BaseModel を生成する layer を挟むか
  - 直接派生だと Pydantic の API がそのまま使える（`.model_validate`, `.model_dump`）
  - 隔離するとカスタマイズの自由度が高い
  - → ⚖️ 議論余地
