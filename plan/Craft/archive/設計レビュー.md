## 1. 設計の強み（残すべき判断）

✅ **`@component` のボイラープレート削減** — これは本質的価値。50→15 行は数値として明快。
✅ **動的スキーマで Web 追加 → veriq 認識** — PoC として既に動いている、貴重な実証。
✅ **API ファースト + Swagger UI** — Web を後回しにする判断と整合的。
✅ **veriq との密結合方針** — 抽象化コストを払わない判断は健全（[[対処方針]] §C.5）。
✅ **`@analysis` 方針 C 確定** — 段階導入計画も明示、過剰なジェネリック化を避けている。
✅ **`@query` で横断検索のカタログ化** — registry パターンの一貫した適用。

## 2. 矛盾・不整合（要修正）

### 2.1 🔴 veriq への態度が場所ごとに違う

- [[対処方針]] §C.5: **「抽象化しない、密結合で割り切る」**
- [[API設計]] §veriq Pass-through API: **subprocess で薄くラップ**して REST 化（疎結合的）

→ 二つは矛盾ではないが、**「内部実装は密結合、外部公開は薄いラッパ」と明示的に区別**する一文が要る。
→ また Pass-through API は **subprocess 起動コスト** が無視できない（毎リクエスト数百ms）。同一プロセス内で `veriq.cli.main` を import する方が現実解。これも [[API設計]] に明記すべき。

### 2.2 🔴 「Web 後回し」と「Web 向け endpoints 一覧」の矛盾感

- [[API設計]] §Web 時代に追加される endpoints で **draft / SSE / per-field validation** を詳細に書いている
- これは「**想定しておく**」のは正しいが、**実装スコープに入れる**ように読める書き方
- → 「**Phase 2 タグ**」を全項目に明示、Phase 1 では NoOp / 設計のみ、と切り分ける

### 2.3 🟡 paths.py vs マルチプロジェクト

- [[設計書]] §2: `paths.py` が単一参照点（single project 前提）
- [[API設計]]: 全 endpoint に `/projects/{pid}/` を入れる（マルチ前提構造）

→ 整合させるには:
- **Phase 1**: `{pid}=default` 固定で受け取り、内部は `paths.py` に解決
- **Phase 2**: `Project(data_root)` クラスに移行、`paths.py` は廃止 or default 解決のみ

これを設計書側にも明記する必要がある。

### 2.4 🟡 registry の数が増えすぎている

現在登場するもの:
1. `ComponentRegistry` ([[コンポーネントデコレータ仕様]])
2. `AnalysisRegistry` ([[API設計]] §Analysis)
3. `QueryRegistry` ([[API設計]] §Cross-component Query)
4. veriq の内部スコープ管理

→ 4 つあるが、本質的には **「decorator で登録するもの一覧」** という同じ概念。
→ **統一 registry** にして `kind={"component", "analysis", "query"}` で分けるか、最低でも **共通 base class** を抽出すべき。バラバラだとテストフィクスチャ・auto_discovery・stub 生成を 3 回書く羽目になる。

### 2.5 🟡 `@analysis` の入力解決方法が未定義

- [[API設計]] §Analysis では `{"spec": {"$ref": "components/power/battery/main"}}` と書いた
- が、`$ref` 解決時の **エラー処理** / **権限** / **多段参照** が未定義
- 単純に文字列パスで良い? `"@components/power/battery/main"` のような sigil?
- → spec として固める前に **記法を 1 つに絞る** べき

## 3. 抜け（補完が必要）

### 3.1 🔴 依存方向図の更新漏れ

[[設計書]] §2 の依存方向図には **analysis / query 層が登場していない**。
追加後は:

```
verification ─┐
backend      ─┤
              ├─► query ─────┐
              ├─► analysis ──┤
              │              ├─► schema ──► registry
              └─► paths      │
                  core ──────┘
```

このように再整理が必要。

### 3.2 🔴 エラーレスポンスの統一仕様が無い

- [[API設計]] §設計原則 6 で「RFC 7807 風」と言及したが具体例なし
- 全 endpoint で **どんなエラー JSON を返すか** のスキーマを 1 ヶ所に集約すべき
- 422 (validation) / 409 (conflict) / 412 (etag mismatch) / 404 / 500 のサンプルを並べた節を追加

### 3.3 🔴 `gen_stubs` の実装スコープ未定義

- [[対処方針]] §A.1 で「`.pyi` auto-generate」を最優先と言ったが:
  - 何の Python 表現から、何を出力するのか
  - 出力先の import パスは固定か可変か
  - `@analysis` / `@query` の stub も同じ仕組みで出すのか（一貫性）
  - **未実装の機能を「最優先」と書いている割に詳細がゼロ**

→ 別ノート [[型基盤設計]] を切って、`gen_stubs` の入出力 / 配置 / CI 連携を具体化すべき。

### 3.4 🟡 非同期 job システムの設計欠落

- [[API設計]] §Verification と §Analysis の両方が **「非同期 job ID 返却」** を要求
- が、job ストレージ / TTL / cancellation / progress / 失敗時挙動が全部空白
- 専用ノート [[Job実行基盤]] が必要。最小実装でも **(a) in-memory dict** か **(b) `runs/jobs/{id}.json` ファイル** かの方針決定が要る

### 3.5 🟡 LLM 連携の具体プラン欠落

- 元々ユーザーが「概念設計の自動化」を最終目標として言及
- [[設計書]] §6.3 と [[懸念事項]] §2.6 で軽く触れているが **設計レベルでの言及がない**
- 「JSON Schema を tool definition として LLM に渡す」「`Entry.model_validate()` を constrained output に使う」あたりは、API ファースト方針と相性が良い
- → [[LLM連携設計]] を Phase 2 候補として明示

### 3.6 🟡 セキュリティ視点（subprocess injection）

- [[API設計]] §veriq Pass-through API で subprocess を使うなら、 **path / node_id / scope** がユーザー入力 → コマンドラインに乗る
- shell injection / path traversal を防ぐホワイトリスト検証が必要
- 単一ユーザー localhost でも書いておくべき安全策

### 3.7 🟢 ドキュメント自動生成

- [[懸念事項]] §3.5 で言及して以降、具体化なし
- registry から自動 markdown 生成 → Obsidian に貼る運用は **本ノート群との接続点** として明示価値あり

## 4. 過剰設計の疑い（YAGNI 候補）

### 4.1 🟡 Phase 1 で `/projects/{pid}` 必須は本当に必要?

- 単一プロジェクト前提なら **path に入れない方が** RESTful にシンプル
- 「Phase 2 で URL 設計を変える」のは破壊変更だが、 **その時に v2 prefix で破壊する** 選択肢もある
- **代替**: Phase 1 は `/api/...` 直書き、Phase 2 で `/api/v2/projects/{pid}/...` を生やす
- → 議論余地あり。今の "default 固定" 設計は **保険として悪くない** が、無料ではない

### 4.2 🟡 ETag / 楽観的ロックは Phase 1 で必要?

- [[対処方針]] §C.1 で導入を推奨したが、**個人開発機 + 1 ユーザ** なら現実に競合は起きない
- asyncio.Lock だけで十分かもしれない
- → ETag は **「自分が作りたい品質」と「過剰さ」のバランス**。 **「実装はするが必須化しない」** 落とし所がありえる
  - GET レスポンスに ETag は付ける（無料）
  - PUT/PATCH/DELETE で `If-Match` を **任意** に（送られたら検証、無ければスキップ）

### 4.3 🟡 schema migration を Phase 1 でやる必要?

- [[対処方針]] §A.4 で `schema_version` + migration scripts を提案
- が、**まだ v1 しか無い** 段階で migration 機構を整備するのは典型的な YAGNI
- → **`schema_version` フィールドだけ Phase 1 で入れておく**（書き込みだけ）。migration コードは **2 つめのバージョンが必要になったら書く**

### 4.4 🟡 全 sub-decorator (`cf`/`af`/`qf`) は本当に分ける必要?

- 三つとも `pydantic.Field` を返すラッパ
- 違いは **意味付け** (`component field` / `analysis field` / `query field`)
- → **`fld()` 1 つに統合** し、引数で意味を渡す方が学習コスト低い
- 反論: 意味別の方が型チェック / IDE 補完で挙動を変えやすい
- → **議論の余地あり**。決断を [[コンポーネントデコレータ仕様]] に追記すべき

### 4.5 🟢 自動 git commit (per API write)

- [[対処方針]] §D.1 で提案したが:
  - log が汚れる
  - 「設計者の意図」を表す commit message が機械生成では弱い
  - 競合・rollback 時にむしろ git 履歴が混乱
- → **手動 commit 推奨 + アプリ層に JSONL audit log** の方が現実的かも
- 自動 commit は **opt-in** にする

## 5. 命名・規約の歪み

### 5.1 ヘルパー名の不統一

| 現状 | 用途 |
|---|---|
| `cf()` | component field |
| `af()` | analysis field |
| `qf()` | query field |

→ どれが何だったか覚えられない。 **`field()` か `fld()` に統一** が無難（後述）。

### 5.2 「subsystem」「component」「instance」の用語

- `subsystem` = `power`, `thermal`, ...
- `component` = `Battery`, `SolarPanel`, ... (型)
- `instance` = `main`, `aux`, ... (実体)

→ これ自体は OK。ただし **エンドポイント名で `components/...` がインスタンス操作を含む** のはやや不正確。
- 純粋に正しくは: `/api/.../{subsystem}/{component_type}/{instance_name}`
- 「component」を **「型」と「インスタンス」両方の意味で使い分けている** ことを用語集に明記すべき

### 5.3 Phase の境界が曖昧

- Phase 1 / 1a / 1b / 2 / 3 が混在
- → 用語集ノート [[フェーズ定義]] か [[Craft]] にスプリント計画として一覧化

### 5.4 日本語 / 英語混在

- ノート本文は日本語、コード例・API path は英語 → これは正常
- が、`desc=`/`description=` などフィールドメタデータの言語ポリシーが未定義
- → 「コード上は英語 desc、UI 表示時にローカライズマップを別途持つ」など方針を [[コンポーネントデコレータ仕様]] に追記

## 6. アーキテクチャ的に考え直したい点

### 6.1 「decorator 過多」になっていないか

現在 / 想定の decorator:
1. `@component` (本プロジェクト)
2. `@analysis` (本プロジェクト)
3. `@query` (本プロジェクト)
4. `@scope.calculation` (veriq)
5. `@scope.verification` (veriq)
6. `@scope.root_model` (veriq)

→ 6 種類は **学習コストが高い**。せめて 本プロジェクト 3 種は **同じ規約**（引数名 / kind 引数 / 戻り型変換ルール）で統一。

提案: **`@register(kind="component"|"analysis"|"query", ...)` 1 つに統合**
- 利点: 学習コスト・テスト・stub 生成・auto_discovery を 1 本化
- 欠点: 引数チェックが kind ごとに分岐 → 実装複雑化、IDE 補完が弱まる
- → トレードオフ。 **個別 decorator のまま、共通基底クラスとヘルパーで重複排除** が現実解か

### 6.2 「registry シングルトン」は本当に問題か

- [[対処方針]] §A.3 で「クラスインスタンス化 + fixture」を推奨したが:
  - auto-discovery が module import 副作用で動く現状、 **fixture で差し替えても import 順序の影響が残る**
  - 完全なテスト隔離には **モジュール再 import** が必要 → そもそも難しい
- → 「シングルトンを認め、テストは **`register / unregister` の単体テスト** で代替」もあり

### 6.3 `auto_discover` の代替

- `pkgutil.iter_modules()` は **import 副作用** に依存する古典的 Python パターン
- が、installed package と src layout で挙動が違う、テスト時に副作用が累積する、など罠が多い
- → **明示的レジストリ** (`subsystems = [power, thermal, com]` を 1 ヶ所に書く) の方がデバッグしやすいかも
- 反論: ファイル追加だけで動くという DX 価値が高い

## 8. 全体所感

**良い点**:
- 「Web 後回し、API ファースト」の判断が首尾一貫
- registry + decorator パターンが本プロジェクトの背骨として通っている
- veriq 公式 CLI を踏まえた API 設計になっている

**懸念**:
- **抽象の層が早くも 3 つ**（component / analysis / query）+ veriq で 6 種類の decorator → **「PoC のはずが framework を作り始めている」リスク**
- Phase 1 のスコープが **「やりたいこと全部」** になりつつある → 削れるところは削る覚悟
- ノート間で **同じ概念を別の言葉で書いている箇所** が出始めている（このレビューで指摘した分以外にも要洗い）

**処方箋**: 次のセッションで [[フェーズ定義]] を作り、 **「Phase 1 で動かすもの」のスコープを明示的に絞る** ことを推奨。具体的には **「power サブシステムだけで `@component` + `@analysis` + `@query` + Swagger」が動く** を Phase 1 ゴールに定義、それ以外は Phase 2 へ追い出す。
