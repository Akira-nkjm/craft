## 1. LLM 連携の具体実装

### そもそも何の話か

このプロジェクトの最終目標の 1 つは「自然言語の要求から **概念設計を自動提案** させる」こと。
例えば:

> 「3 年間の地球低軌道ミッション、観測機器の消費電力は平均 20W、データ取得は 1 日 1GB」

と打ち込んだら、LLM が:
- 必要なバッテリー容量を計算
- 太陽電池の枚数を提案
- 通信機の規格を選択
- → これらを TOML エントリとして書き出し
- → veriq に検証させて、不合格なら直して再提案

というループを回したい。
**この LLM とのやり取りをどう実装するか** が論点。

### LLM 連携の 3 つの方式

#### (A) OpenAI Function Calling / Tools

GPT-4 や o1 などに「**この関数があるよ**」と JSON Schema を渡しておくと、
LLM が必要に応じて「この引数で関数を呼びたい」と応答してくる。

```python
tools = [{
    "name": "add_battery",
    "description": "バッテリーを追加する",
    "parameters": BatteryEntry.model_json_schema()   # ← 我々の registry から
}]
response = openai.chat.completions.create(model="gpt-4o", tools=tools, ...)
# response から "add_battery({...})" の呼び出し要求が返る
```

- **長所**: 業界標準、ライブラリが豊富
- **短所**: OpenAI 専用、価格はやや高い

#### (B) Anthropic Tool Use

Claude (Anthropic) の同等機能。形式が少し違うだけ。
**Claude Code を使っている**こと、推論能力の高さから本プロジェクトには相性が良い。

#### (C) MCP (Model Context Protocol)

Anthropic 発の **「LLM クライアントと外部ツールをつなぐ標準プロトコル」**。
我々のシステムを MCP サーバとして公開すると、Claude Desktop / Claude Code から **直接** TOML 操作や検証が叩ける。

```
[ Claude Code / Desktop ] ──MCP──► [ 我々の MCP サーバ ] ──► registry / TOML / veriq
```

- **長所**: ユーザが Claude Code を開いた瞬間に対話できる、開発者体験が劇的に良い
- **短所**: まだエコシステムが新しい、OpenAI 等とは別途連携が必要

### propose ループとは

LLM に **「提案 → 検証 → フィードバック → 再提案」** を繰り返させる仕組み:

```
1. ユーザ要求を LLM に渡す
2. LLM が候補 TOML を生成
3. registry の Pydantic で validate
   - 失敗 → エラーを LLM に返して再生成（goto 2）
4. TOML に書き込み、veriq calc 実行
5. 検証失敗の要求があれば LLM に返して修正案を求める（goto 2）
6. 全要求充足 → 完成
```

このループの **状態管理**（途中までの提案、過去の失敗、現在の検証結果）をどう持つか、
**コスト管理**（無限ループ防止、トークン消費上限）をどう実装するかが論点。

### コスト管理の現実

- GPT-4o: 約 $5/M input + $15/M output tokens
- Claude Opus: 約 $15/M + $75/M
- **1 ループあたりおよそ $0.1〜$1**、10 ループで $1〜$10
- 上限なしで放置するとうっかり数十ドル飛ぶ → **必須の安全装置**

### 推奨案

✅ **MCP を Phase 1 で提供 + Anthropic Tool Use の propose ループは Phase 2 以降**

- まず MCP サーバとして公開すれば、ユーザは Claude Code から手動で対話できる
- 「自動 propose ループ」は本気の研究テーマなので、土台が固まってから
- MCP サーバを書くこと自体は数百行で済む（FastAPI の registry を MCP に貼るだけ）

## 2. 非同期 job runner の実装

### そもそも何の話か

**「時間のかかる処理を、HTTP リクエストとは別に裏で走らせる」** 仕組み。

例: `POST /api/verify` で全サブシステム検証を走らせると **10秒〜数分** かかる。
HTTP リクエストでそのまま待つと:
- ブラウザ / curl がタイムアウト
- 進捗が見えない
- 並列に他の操作ができない

→ **「ジョブを受け付けて ID を返す → 別途進捗確認」** の形が標準解:

```
POST /api/verify              → 202 Accepted, { "job_id": "abc123" }
GET  /api/jobs/abc123          → { "status": "running", "progress": 0.3 }
GET  /api/jobs/abc123          → { "status": "done", "result": {...} }
```

### 実装の選択肢（軽量→重量）

#### (A) Python dict（プロセス内）
```python
jobs: dict[str, JobState] = {}
asyncio.create_task(run_verification(job_id))
```
- **長所**: ゼロ依存、即実装可能
- **短所**: サーバ再起動で消える、別プロセスから見えない、並列度はプロセス内 thread / async に限定

#### (B) ファイルベース（`runs/jobs/{id}.json`）
ジョブ状態を JSON ファイルに書く。
- **長所**: 再起動後も残る、外部ツールで覗ける
- **短所**: 同時更新の競合、ファイル数の管理

#### (C) SQLite
組み込み DB。1 ファイルで完結。
- **長所**: 永続化、クエリ可能、トランザクション安全、Python 標準
- **短所**: スキーマ管理、`asyncio` との相性に注意

#### (D) 既存ライブラリ (rq / dramatiq / Celery)
本格的なジョブキューライブラリ。Redis や RabbitMQ が必要。
- **長所**: スケーラブル、再試行 / 優先度 / cron 機能
- **短所**: インフラ依存、ローカル開発が複雑化

### 推奨案

✅ **(B) ファイルベース + asyncio.Task**

- Phase 1 の規模（個人ユース、せいぜい同時数 jobs）なら十分
- 「過去の run 結果」を `runs/jobs/{id}.json` に残す方針は既に [[対処方針]] にある → 統一できる
- SQLite に上げるのは **「ジョブが 100 件超え始めたら」** で良い
- Celery 系は **絶対に不要**（このプロジェクトの規模ではオーバーキル）

## 3. git audit と JSONL audit の関係

### そもそも何の話か

**「誰が・いつ・何を変えたか」** を記録する仕組み = audit log。
本プロジェクトでは 2 つの記録方法を提案している:

#### git audit
TOML の変更ごとに自動 `git commit`。
```
6d716b2 feat(power): add batteries.aux via API
ffe94aa fix(power): update batteries.main.capacity_wh 100 → 120
```

#### JSONL audit
専用ログファイルに JSON Lines 形式で追記。
```json
{"ts":"2026-05-22T10:00:00","action":"create","path":"power/battery/aux","user":"akira"}
{"ts":"2026-05-22T10:05:00","action":"update","path":"power/battery/main","diff":[...]}
```

### それぞれの長所短所

| | git audit | JSONL audit |
|---|---|---|
| 永続性 | 強（分散・履歴改竄困難） | ファイル削除可能 |
| 復元 | `git checkout` で簡単 | 自前で実装 |
| 履歴閲覧 | `git log` / `gh` / GitHub UI | `tail`/`jq`/`grep` |
| 機械的解析 | コミットメッセージ parsing 必要 | JSON で直接読める |
| 検索性 | 弱い（コミット全文走査） | 強い（grep / jq） |
| ノイズ | log が高頻度で汚れる | 別ファイルなので git は綺麗 |
| 「誰が」 | git config / 環境変数 | アプリ層で持つ user |

### 両者の役割の違い

- **git audit** = **「データの状態」** の履歴。「あの時の TOML はどうだったか」を取り出せる
- **JSONL audit** = **「操作の流れ」** の履歴。「いつ何のリクエストが来たか」「どんなエラーが出たか」

→ **本質的には別の情報を記録している**。両方持つのが正解。

### 推奨案

✅ **両方持つ。役割を明確に分ける**

- **JSONL は常時 on**（軽量、構造化、検索しやすい）
- **git auto-commit は opt-in**（環境変数 or config で）
  - 通常は **手動 commit** 推奨（人間が意味あるメッセージを書く）
  - 自動化したいケース（バッチ実行、LLM agent 運用）でのみ on
- データの誤削除復元は **JSONL に diff を含めれば** git なしでも可能

## 5. doc 自動生成の配置

### そもそも何の話か

registry に登録された component / analysis の **一覧と仕様** を markdown で自動生成して、
人間が読めるドキュメントとして提供する仕組み。

例:
```markdown
## Components

### Battery (power)
- capacity_wh: float, Wh, "Battery capacity"
- nominal_voltage_v: float, V, default 0.0
...

## Analyses

### total_power_by_mode (power)
"OperationMode 別の消費電力合計"
Inputs:
  - batteries: vq.Table[BatteriesName, BatteryEntry]
  - mode: OperationMode
Output: float
```

### 配置場所の選択肢

#### (A) Obsidian vault に直接書き込む
`02_Dev/Projects/Craft/generated/` 配下に出力。
- **長所**: Obsidian でそのまま閲覧、wiki-link で本ノートと相互参照可能
- **短所**: 自動生成物が vault を肥大化、誤って手で編集して上書きされる事故

#### (B) リポジトリ内 `docs/generated/`
コード本体と同じリポジトリに出力。
- **長所**: コードと一緒にバージョン管理、CI で生成
- **短所**: Obsidian と分断、両方見る必要

#### (C) Web で動的生成（Swagger UI と同じ場所）
FastAPI の `/docs` 周辺で見れるように。
- **長所**: 常に最新、静的ファイル不要
- **短所**: API サーバ起動が必要、Obsidian 内では見れない

### 推奨案

✅ **(B) + (C) の併用**

- **(C) FastAPI で動的提供**: `/docs/components` 等。常に最新、開発時の主参照
- **(B) `docs/generated/` 出力**: CI で生成、GitHub で読める。リファレンス的に
- **(A) Obsidian 直書きはやらない**: vault には [[設計書]] / [[Ref設計]] など **人間が書く** ものだけ置く。**自動生成物と手書きを混ぜない**のが鉄則

## 6. veriq Python API の安定性

### そもそも何の話か

[[API設計]] の **veriq Pass-through API** で、`veriq calc` 等を呼ぶ方法に 2 案ある:

#### (A) subprocess（外部プロセスとして起動）
```python
subprocess.run(["veriq", "calc", "project.py", "-i", "in.toml", "--json"])
```
- **長所**: シンプル、CLI と完全に同じ挙動が保証される
- **短所**: 毎回 Python インタプリタ起動 **数百ms〜数秒**、入出力は文字列のみ

#### (B) Python API として直接 import
```python
from veriq.cli import main as veriq_main
veriq_main(["calc", "project.py", "-i", "in.toml"])
# あるいはより低レベル
from veriq import Project, run_calc
result = run_calc(project, input_data)
```
- **長所**: 高速（プロセス起動なし）、エラーを Python 例外で受け取れる、Python オブジェクト直接渡し
- **短所**: **veriq が公開 API として保証しているか不明**。内部実装に依存するとバージョンアップで壊れる

### 「公開 API」と「内部 API」の違い

Python ライブラリは **「これは外から使っていい」** と作者が宣言した関数群（=公開 API）と、
**「内部実装、変更する可能性あり」** の関数群がある。

- 公開 API: ドキュメントに載っている、`__all__` に含まれる、関数名がアンダースコアで始まらない
- 内部 API: `_func()` のようなアンダースコア prefix、ドキュメント記載なし

内部 API に依存すると **次バージョンで突然動かなくなる** リスク。

### 推奨案

✅ **暫定 subprocess、Phase 2 で公式 API 確認後に切替**

- Phase 1a: subprocess で動かす（簡単・確実）
- 並行して: veriq のソースコードを読んで **公開 API として呼べる関数を特定**
  - もしくは veriq 作者に「Python API として叩きたい」と相談する手も
- Phase 1b: 公開 API が見つかれば直接 import に切替（数百ms × N リクエスト = 大きな改善）

## 7. TOML スキーマレス領域

### そもそも何の話か

現在 `extra="forbid"` で **登録されていないフィールドを全部エラー** にしている。
これは typo 検出には強いが、**逆に困るケース**:

```toml
[batteries.main]
capacity_wh = 100
manufacturer = "Panasonic"

# ↓ こういうメモを書きたいけどスキーマにない
notes = "選定理由: 過去ミッションでの実績、価格優位性"
datasheet_url = "https://..."
reviewed_by = "akira"
review_date = "2026-05-22"
```

これらは **registry 管理外** だが、ユーザは TOML に残しておきたい。

### 選択肢

#### (A) `extra="forbid"` 緩めて `extra="ignore"` に
未知フィールドは黙って無視。
- **長所**: 何でも書ける
- **短所**: typo も無視される（`capcity_wh = 100` がエラーにならない）← 致命的

#### (B) 専用のメタデータ欄を用意
全 component に固定で `_meta` フィールド:
```toml
[batteries.main]
capacity_wh = 100
[batteries.main._meta]
notes = "..."
datasheet_url = "..."
```
- **長所**: 領域が明示的、typo は本体側で引き続き検出
- **短所**: ユーザに「`_meta` に書いてください」と覚えてもらう

#### (C) 任意の `tags: dict[str, Any]` を許可
```toml
[batteries.main]
capacity_wh = 100
[batteries.main.tags]
notes = "..."
```
- (B) と同じ発想、別名

#### (D) 別ファイル分離
`power.toml` (厳格) + `power.notes.toml` (自由)
- **長所**: 完全分離、責務明確
- **短所**: ファイル数増、対応関係を維持する手間

### 推奨案

✅ **(B) `_meta` フィールド方式**

- 全 component の Pydantic に `_meta: dict[str, Any] | None = None` を自動付与
- ドキュメントで「自由メモは `_meta` に」と明示
- 厳格性 (typo 検出) を維持しつつ柔軟性を提供
- **`extra="ignore"` には絶対にしない**（型基盤の価値が消える）

## 8. テストの「現実 TOML」依存度

### そもそも何の話か

テストコードで TOML データを使うとき、 **本物のプロジェクト TOML を使うか、テスト専用の小さい TOML を作るか** という選択。

### 2 つのアプローチ

#### (A) リアル TOML をテストで使う
```python
def test_battery_load():
    data = load_toml("design/data/system/power.toml")
    assert "main" in data["batteries"]
```
- **長所**: 本番と同じデータで検証 → 現実的なバグを捕まえる
- **短所**:
  - データ変更でテストが壊れる（flaky）
  - 大きい TOML だとテスト遅い
  - テストデータの意図が読みにくい

#### (B) fixture でテスト専用 TOML を作る
```python
@pytest.fixture
def sample_power_toml(tmp_path):
    p = tmp_path / "power.toml"
    p.write_text('[batteries.main]\ncapacity_wh = 100\n')
    return p

def test_battery_load(sample_power_toml):
    data = load_toml(sample_power_toml)
    assert data["batteries"]["main"]["capacity_wh"] == 100
```
- **長所**: テストの意図が明確、データ変更に強い、高速
- **短所**: 「現実の複雑さ」を模さない → 本番だけで起きるバグを見落とす

### テストの層別の使い分け

| テスト層 | 推奨アプローチ |
|---|---|
| Unit | (B) 完全 fixture。小さく速く |
| Contract (snapshot) | (B) fixture |
| Integration | (B) fixture、ただし「現実っぽい」サイズ |
| Property | (B) Hypothesis でランダム生成 |
| **E2E** | **(A) リアル TOML を含む** smoke test |

### 推奨案

✅ **層別運用 + 「snapshot TOML」 戦略**

- 単体・統合テストは fixture 中心 (B)
- **代表的な実 TOML を `tests/snapshots/` にコピー** して「これがロード・検証通る」だけは保証する E2E test を 1 つ持つ
- 実 TOML が更新されたら snapshot を更新（人間が意図的に同期）
- これで両方の利点（速さ + 現実性）を取れる
