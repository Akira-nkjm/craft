## 2. アーキテクチャ全体像

### 依存方向

```
verification ─┐
backend      ─┼──► schema ──► registry
              │              ▲
              └─► paths      │
                  core ──────┘
```

`paths.py` がデータルートの単一参照点。`verification → backend` の逆依存を排除。

### レイヤ構成

| レイヤ | 役割 | 主なファイル |
|--------|------|-------------|
| `schema/` | 型定義（`@component`、registry、root model builder） | `_component.py`, `_registry.py`, `_root_model_builder.py` |
| `core/` | データ I/O（TOML 読み書き、動的 enum） | `reader.py`, `writer.py`, `dynamic_enum.py` |
| `backend/` | FastAPI サーバー | `main.py`, `routers/` |
| `frontend/` | Vanilla JS SPA | `index.html` |
| `verification/` | veriq 検証スコープ | `project.py`, `scopes/power.py` |

## 3. コアコンポーネント

### 3.1 `@component` デコレータ

クラス定義から Pydantic モデル群を自動生成し、レジストリに登録。

```python
@component(mixin="temperature")
class Battery:
    capacity_wh: float = cf(ge=0, unit="Wh", desc="Battery capacity")

    class Design:
        depth_of_discharge: float = cf(ge=0, le=1)

    class Requirements:
        depth_of_discharge_max: float = cf(default=0.8, gt=0, le=1)
```

**生成物**: `BatterySpec` / `BatteryDesign` / `BatteryRequirements` / `BatteryEntry` + registry 登録。

**オプション**:
- `base="spec" | "power_spec"`
- `mixin="temperature"`
- `has_power_mode=True`
- `plural="..."`（TOML テーブル名上書き）

**設計判断**: `_strictify()` で全モデルに `extra="forbid"` を一律適用 → typo を Web / TOML 直書き両方で早期検出。

### 3.2 ComponentRegistry

シングルトン。registry が全レイヤの単一参照点:
- ルーター: JSON Schema 配信、CRUD バリデーション
- root model ビルダー: veriq 互換 root の動的生成
- veriq スコープ: 動的 enum + entry model のペア取得

### 3.3 サブシステム auto-discovery

`schema/subsystems/__init__.py` で `pkgutil.iter_modules()` 走査。新ファイル追加で自動登録、ルーター/ビルダー/スコープの編集不要。

### 3.4 動的 StrEnum (`load_instance_enum`)

TOML の現在キーから StrEnum を import 時に構築。
→ **「Web 追加後に veriq が即認識する」仕組みの核心**。

### 3.5 動的 root model (`build_subsystem_root_model`)

registry + TOML の現在状態から veriq 互換 root を生成。`vq.Table[dyn_enum, entry]` を組み立て、新コンポーネント追加時の root model 編集を不要化。

### 3.6 アトミック TOML 書き込み

`tempfile → flush → fsync → os.replace` のパターン。POSIX 上で同一 FS 内ならアトミック。並行リーダーは truncate された TOML を見ない。

## 4. データフロー

### Web からのインスタンス追加

```
[Web フォーム]
  ├── GET  /api/schema/power/battery
  │        → BatteryEntry.model_json_schema()
  │        → フォーム自動生成 + バリデーション
  └── POST /api/components/power/battery/aux
           → BatteryEntry.model_validate(body)   (422 on error)
           → writer.add_instance(...)
           → atomic write to power.toml
```

### veriq 検証 (CLI)

```
uv run python -m veriq calc ...
  → craft.verification.project import
  → schema.subsystems import (auto-discovery)
  → registry に Battery/SolarPanel/... 登録
  → load_instance_enum(toml, "batteries")
  → build_subsystem_root_model("power", root)
  → 追加済みインスタンスを含めて検証
```

## 5. 重要な設計判断 / 落とし穴

### 5.1 `from __future__ import annotations` は禁止

veriq は `@scope.calculation()` で `inspect.signature()` を使い、引数アノテーションをライブオブジェクトとして検査する。`__future__` で文字列化すると `vq.Ref()` が解決されず `TypeError`。

### 5.2 動的 enum はプロセス単位で凍結

`scopes/power.py` の module import 時に StrEnum と root model が確定する:
- ✅ CLI 実行（毎回 fresh プロセス）→ OK
- ❌ FastAPI 内 in-process verify → 追加した key が enum に反映されない
- → Phase 2 で `build_project(data_root)` factory 化

### 5.3 既存スタックと共存

`design/` の `BaseSpec` / `BaseDesign` / `ComponentEntry` をそのまま再利用。`design/data/subsystem/*.toml` を共有。既存の `design/project.py` の veriq 実行は無影響。

## 6. 共同検討メモ

> ここから一緒に深掘りしていく領域

### 6.1 `@component` 表現力の拡張

現状の限界:
- **datasheet_only**（`design` を持たない）: `SolarCell`、`MaterialEntry`、`SurfaceMaterialEntry`
- **ネスト config**: `LinkBudget`（`bit_rates.{low,high}`、`coding_gain.{convolutional,turbo}`）

検討:
- `@component(kind="datasheet")` でスキップ?
- ネストは inner class を再帰的に処理?

### 6.2 サブシステム間の依存と検証順序

power → thermal の熱負荷、com → power の消費電力など、サブシステム横断の検証をどう表現するか。

### 6.3 LLM による概念設計の自動化

「概念設計の自動化」を最終目標とするなら、registry の型情報 + 既存 TOML の参照例を LLM に与えて新インスタンスを提案させる経路が自然。

- 入力: 要求仕様（自然言語）
- 出力: 候補コンポーネントの TOML エントリ
- 検証: veriq で要求充足を確認 → フィードバックループ

### 6.4 移行ロードマップ

| Phase | 内容 |
|-------|------|
| 1 (完了) | `power` の `@component` 化、Web 追加 → veriq 認識を実証 |
| 2 | `@component` 表現力拡張、`build_project` factory 化、in-process verify |
| 3 | `com` / `thermal` の veriq スコープ実装 |
