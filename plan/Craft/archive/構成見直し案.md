## Q1. `system=` をディレクトリから自動推論できるか

### 結論: ✅ できる、自動推論を default にする

ファイル `systems/power/components.py` から `@component` を呼ぶ時、`inspect.stack()` で呼び出し元ファイルパスを取れる:

```python
def component(system: str | None = None, ...):
    if system is None:
        # 呼び出し元 file から推論
        caller_path = Path(inspect.stack()[1].filename)
        # systems/power/components.py → "power"
        system = _infer_subsystem_from_path(caller_path)
    ...
```

### 推論ルール

```
.../systems/<name>/...   → system = "<name>"
.../systems/<name>.py    → system = "<name>"
それ以外                      → system 明示必須、無ければエラー
```

### 明示優先

```python
@component()                         # 自動推論
class Battery: ...

@component(system="power_alt")    # 明示で上書き
class Battery: ...
```

→ デフォルトは推論、テスト・特殊ケースは明示。

### Pros / Cons

| Pros | Cons |
|---|---|
| DRY、定型コード削減 | 「magic」が増える |
| 移動・リネームに強い | テスト fixture で別 dir に置きにくい |
| `system` の typo 起源バグが消える | スタックフレーム解析のコスト（無視できる） |

→ **Pros > Cons**。Cons は **明示モード**で逃げられる。

## Q2. `base=` / `mixin=` をすっきりさせる方法

### 現状の不満

```python
@component(system="power", base="power_spec", mixin="temperature", has_power_mode=True)
class PDM: ...
```

decorator 引数が文字列ばかりで増える → 読みづらい、IDE 補完が効きにくい、typo に弱い。

### 検討した案

#### 案 A: trait/marker class 多重継承 ✅ 採用

```python
from craft.schema import component, fld
from craft.schema.traits import PowerConsuming, TemperatureSensitive, HasPowerMode


@component
class PDM(PowerConsuming, HasPowerMode):
    rated_current_a: float = fld(ge=0, unit="A")

    class Design:
        efficiency: float = fld(ge=0, le=1, default=0.95)


@component
class SolarPanel(PowerConsuming, TemperatureSensitive):
    area_m2: float = fld(ge=0, unit="m^2")
    efficiency: float = fld(ge=0, le=1, default=0.28)

    class Design:
        cell_count: int = fld(ge=1)


@component
class Battery(TemperatureSensitive):
    capacity_wh: float = fld(ge=0, unit="Wh")
    # ...
```

**仕組み**:
- `PowerConsuming` / `TemperatureSensitive` / `HasPowerMode` などは **マーカークラス**
- `@component` が `cls.__mro__` を走査して、各マーカーに対応するフィールド・基底を組み立てる
- マーカー自体は空クラス or `Protocol`、実体は decorator 側

**Pros**:
- decorator 引数 0〜1 個に減る（system は自動、それ以外は基底クラスで）
- `from .traits import ...` で **IDE 補完・hover** で意味が見える
- typo 不可能（クラス未定義は import エラー）
- 多重トレイト合成が **class 構文で自然**
- pyrefly でも型として認識される

**Cons**:
- `@component` 実装がやや複雑（trait → field 変換ロジック）
- マーカークラスを「飾り」と理解する必要

#### 案 B: inner Config class
```python
@component
class PDM:
    class _Config:
        traits = [PowerConsuming, HasPowerMode]
    rated_current_a: float = fld(ge=0, unit="A")
```
→ ネストが深い、`_Config` が Pydantic v1 風で紛らわしい。**❌**

#### 案 C: 別 decorator
```python
@power_component(temperature=True, has_power_mode=True)
class PDM: ...
```
→ system ごとに decorator が増える。組み合わせ爆発。**❌**

#### 案 D: 引数を `traits=[...]` リストに統一
```python
@component(traits=[PowerConsuming, HasPowerMode])
class PDM: ...
```
→ 案 A と機能等価だが、class 構文より冗長。**△**

### 確定: 案 A (trait/marker 多重継承)

### Trait の意味マッピング

| Trait | 効果 |
|---|---|
| `PowerConsuming` | Spec に `default_power_consumption_per_unit_w: float` を追加 |
| `PowerGenerating` | Spec に発電関連フィールドを追加 |
| `TemperatureSensitive` | Spec に `operating_temperature_min_c` / `_max_c` を追加 |
| `HasPowerMode` | Design に `power_modes: dict[OperationMode, bool]` を追加 |
| `Datasheet` | Design 自動生成をスキップ（Design なし component を明示） |

→ **`base=` も `mixin=` も `has_power_mode=` も全部 trait に統合**。decorator は `@component` 単体で済む。

### Component シグネチャの最終形

```python
def component(
    cls: type | None = None,
    /,
    *,
    system: str | None = None,    # 自動推論される
    name: str | None = None,         # クラス名 lowercase が default
    plural: str | None = None,       # 自動複数形化
    desc: str | None = None,
    tags: Iterable[str] = (),
) -> ...
```

→ **使う時はほぼ `@component` だけ書けば良い**。

## Q3. ディレクトリを system 中心に再編する

### 現状（並列レイアウト）

```
schema/systems/power.py        ← Battery 等の定義
analyses/systems/power.py      ← 計算関数
verification/scopes/power.py      ← veriq scope
data/system/power.toml         ← インスタンスデータ
```

→ **power のことを調べたい** → 4 箇所探す必要。

### 提案（system 中心レイアウト）✅ 採用

```
systems/
├── power/
│   ├── __init__.py
│   ├── components.py     ← @component 群
│   ├── analyses.py       ← @analysis 群
│   ├── verification.py   ← vq.Scope 定義 + root model
│   └── data.toml         ← インスタンスデータ
│
├── thermal/
│   ├── components.py
│   ├── analyses.py
│   ├── verification.py
│   └── data.toml
│
└── com/ ...
```

### Pros / Cons

| Pros | Cons |
|---|---|
| 1 つの system の全要素が同居 | 「全 component を一覧したい」時は registry 経由になる（が、これは元々 registry の仕事） |
| 新規追加が **ディレクトリ単位**でアトミック | ファイルパスが浅い（`components.py` 等の汎用名が多数） |
| `system` 自動推論と相性最高（`parent.name`） | 既存実装 `experiment/` との差分大 |
| import 時の cohesion が高い（同じ system は隣接） | tests/ も同じ階層を持つか別か論点 |
| `data.toml` が **コードのすぐ隣** | data を git で別途管理したい時に困る場合あり |

→ **Pros > Cons** が明確。今は実装初期で移行コストが低い。

### code data の同居問題

`data.toml` をコード隣に置くべきか:
- ✅ ローカル開発: 隣で見える、編集しやすい
- ⚠️ 本番: data は別 volume にしたい
- → **`data.toml` は default 位置、`paths.py` で override 可** にする
  - `os.environ["SATDESIGN_DATA_ROOT"]` などで切替可能
  - default は `systems/{name}/data.toml`、本番は `/var/satdesign/data/{name}.toml` 等

### project / scope 集約

```python
# systems/__init__.py  (新規)
import veriq as vq

from .power.verification import power
from .thermal.verification import thermal
from .com.verification import com

project = vq.Project("SpacecraftDesign")
project.add_scope(power)
project.add_scope(thermal)
project.add_scope(com)
```

→ project 統合は **systems パッケージの `__init__.py` が担当**。

### 最終的なリポジトリ全体像

```
craft/
├── core/                # 基盤（不変）
```

## 移行後の `power/components.py` 完成形（プレビュー）

```python
"""Power system components."""

from craft.schema import component, fld
from craft.schema.traits import (
    PowerConsuming,
    TemperatureSensitive,
    HasPowerMode,
)


@component
class Battery(TemperatureSensitive):
    """二次電池。eclipse 中の電力供給を担う。"""

    capacity_wh: float = fld(ge=0, unit="Wh")
    nominal_voltage_v: float = fld(ge=0, unit="V", default=0.0)
    manufacturer: str = fld(default="")

    class Design:
        depth_of_discharge: float = fld(ge=0, le=1)

    class Requirements:
        depth_of_discharge_max: float = fld(default=0.8, gt=0, le=1)


@component
class SolarPanel(PowerConsuming, TemperatureSensitive):
    area_m2: float = fld(ge=0, unit="m^2")
    efficiency: float = fld(ge=0, le=1, default=0.28)

    class Design:
        cell_count: int = fld(ge=1)


@component
class PDM(PowerConsuming, HasPowerMode):
    rated_current_a: float = fld(ge=0, unit="A")

    class Design:
        efficiency: float = fld(ge=0, le=1, default=0.95)
```

→ `system=`, `base=`, `mixin=`, `has_power_mode=` がすべて消え、 **本質情報だけが残る**。

## 確定事項

| 項目 | 決定 |
|---|---|
| `system` 自動推論 | ✅ ディレクトリ名から default、明示で上書き可 |
| `base=` / `mixin=` | ❌ 廃止、 **trait 多重継承** に統合 |
| `has_power_mode=` | ❌ 廃止、 `HasPowerMode` trait に |
| ディレクトリ構造 | ✅ **system 中心** (`systems/{name}/{components,analyses,verification,data}.toml`) |
| data の位置 | ✅ コード隣に default、env var で override |
| project 統合 | ✅ `systems/__init__.py` で全 scope を `vq.Project` に追加 |

## 残る論点（次に詰める）

1. **trait の完全一覧** — どの trait をどう実装するか
2. **trait の重複・矛盾検出** — 例: `Datasheet` と `HasPowerMode` を併用したらどうなる
3. **`Datasheet` trait の必要性** — `Design` クラスを書かなければ自動的に「Design なし」になる現状を残すか、明示マーカーを推奨するか
4. **後方互換** — 既存 `experiment/` 配下のコードはどう移行するか（実装段階の課題）
