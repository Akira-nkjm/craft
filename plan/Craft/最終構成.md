## 2. 中心思想 — 「4 種の登場物」+「1 つの registry 観」

設計者が書くものは **4 種類だけ**。**Component / Config は base class（decorator なし）、Analysis のみ decorator**。

| 種類               | 宣言方式                                         | 何を表すか                         | 例                                         |
| ---------------- | -------------------------------------------- | ----------------------------- | ----------------------------------------- |
| **Component**    | `class X(Component):` (default = Singleton)               | hardware 実体（1 機構成）                | OBC, BusStructure, PropulsionModule       |
| **Component**    | `class X(Component, MultiInstance, *traits):`             | hardware 実体（複数積み）               | Battery, SolarPanel, Antenna              |
| **Config**       | `class X(Config):`                           | 非 hardware 設定値（ミッション要求・軌道・環境） | MissionProfile, OrbitalParameters         |
| **Analysis**     | `@analysis def f(...) -> T:`                 | 純粋関数（component を入力に値を返す）      | battery_eol_capacity, total_power_by_mode |
| **Verification** | `@analysis(verify=True) def f(...) -> bool:` | 真偽判定（要求充足）                    | verify_battery_capacity                   |

→ **Component / Config に decorator は付けない**。`@component` 引数あり decorator は採用せず、`Component` 継承の base class 方式（方式 C）を採用。理由は [[コンポーネントデコレータ仕様]] §9（trait の多重継承で文字列引数を型に乗せる、pyrefly 友好）。
→ **Component の default は Singleton**（1 機構成）。複数積む場合は `MultiInstance` trait を明示（[[インスタンス多重度]]）。概念設計の自然な順序「まず 1 機 → 必要に応じて複数化」と整合する。
→ **`@verification` という独立 decorator は持たない**。`@analysis` の特殊ケースとして `verify=True` フラグで表現、内部で veriq の `@scope.verification()` に流す。
→ **`@query` decorator は廃止**（[[Analysis詳細仕様]] §13.4 結論）。横断集計は `@analysis` + `vq.Table` で素直に書く。メタデータ照会は built-in introspection endpoint で。

### `@analysis` の入力参照は veriq `vq.Ref` 流儀

```python
# systems/power/analyses.py （system はファイルパスから自動推論）
@analysis
def total_power_by_mode(
    batteries: Annotated[vq.Table[BatteriesName, BatteryEntry], vq.Ref("$.batteries")],
    altitude: Annotated[float, vq.Ref("$.orbit.altitude_km", scope="orbital")],
    eclipse: Annotated[float, vq.Ref("@calc_eclipse_s", scope="orbital")],
    mode: OperationMode,
) -> float: ...
```

詳細・確定事項は [[Analysis詳細仕様]] §13「引数参照 — vq.Ref 流儀」参照。

### Subsystem Model — veriq Scope の root_model（自動派生）

ユーザが直接書く要素ではないが、**veriq が要求する架橋レイヤ** なので明示する。
veriq の各 Scope は 1 つの **root_model** (`@scope.root_model()` 適用された Pydantic BaseModel) を要求する（[[veriq仕様メモ]] §Core API）。

本プロジェクトでは system ≒ Scope なので、各 system の **Subsystem Model** は registry から **auto-derive**:

```python
# 概念（実装テンプレート systems/power/scope.py が実際に行うこと）
power = vq.Scope("power")

@power.root_model()
class PowerModel(BaseModel):
    batteries:    vq.Table[BatteriesName, Battery.Entry]      # MultiInstance Component
    solar_panels: vq.Table[PanelsName,    SolarPanel.Entry]   # MultiInstance Component
    pdms:         vq.Table[PdmsName,      PDM.Entry]          # MultiInstance Component
    # ↑ registry の power 配下の component / config から自動生成
    # ↑ DynamicEnum は data.toml の現在のキーから build
```

→ **設計者は Component / Config を書くだけ**。`PowerModel` 相当は `build_system_root_model("power")` が registry を走査して動的構築する（[[UnifiedRegistry設計]]）。
→ TOML 階層との対応（veriq 規約 `[<Scope>.model.<...>]` に準拠、[[veriq仕様メモ]] §TOML 入出力の構造）:
  - Singleton Component → `<name>: Component.Entry` → TOML `[<scope>.model.<name>.spec]` `.design` フラット
  - MultiInstance Component → `<plural>: vq.Table[<Plural>Name, Component.Entry]` → TOML `[<scope>.model.<plural>.<key>.<...>]`
  - Config → `<name>: Config.Entry` → TOML `[<scope>.model.<name>]` フラット
  - 計算出力は `[<scope>.calc.<func>.<...>]`、検証出力は `[<scope>.verification.<func>]` に自動配置（[[データパイプライン]] §2.2）

### Registry は **論理的に 1 つ、実装は kind 分割**

```
UnifiedRegistry
  ├── components: {(system, name) → ComponentDefinition}
  ├── configs:    {(system, name) → ConfigDefinition}
  └── analyses:   {(system, name) → AnalysisDefinition}
```

Subsystem Model は registry の **派生物**（永続化されない、起動時に build）。

- `register(kind, definition)` の単一インターフェース
- auto-discovery / stub 生成 / doc 生成 / API 露出 / テストフィクスチャ は全部この登録物を走査
- **「decorator が増えても土台は増えない」**

## 5. 主要モデル / 規約

### 5.1 統一フィールドヘルパー `fld()`

```python
def fld(
    *,
    ge=None, le=None, gt=None, lt=None,
    default=NOT_SET,
    unit: str | None = None,
    desc: str | None = None,
    group: str | None = None,        # UI hint
    order: int | None = None,        # UI hint
) -> FieldInfo: ...
```

→ `cf` / `af` / `qf` は廃止。**1 つ覚えれば全て書ける**。

### 5.2 宣言方式（方式C: base class + decorator）

**Component / Config は base class、Analysis のみ decorator**。
旧 `@component(mixin=..., has_power_mode=...)` の文字列引数は **trait の多重継承** に置き換え（[[コンポーネントデコレータ仕様]] §9 参照）。

```python
# Singleton hardware (default): trait なし
class OBC(Component, PowerConsuming, TemperatureSensitive):
    clock_mhz: int = fld(ge=0)
    class Design: ...

# MultiInstance hardware: MultiInstance trait を付ける
class Battery(Component, MultiInstance, TemperatureSensitive):
    capacity_wh: float = fld(ge=0, unit="Wh")
    class Design: ...
    class Requirements: ...

# Config (非 hardware): Config base class (常 Singleton)
class MissionProfile(Config):
    duration_years: float = fld(ge=0)

# Analysis: decorator のみ残す（関数なので継承が使えない）
@analysis(verify=False, cache=True)
def battery_eol_capacity(...): ...
```

- `system` は **ファイルパスから自動推論**（明示は class 引数 `system="..."`）
- `@query` は廃止（[[Analysis詳細仕様]] §13.4）。横断集計は `@analysis` + `vq.Table` で書く
- 共通引数（analysis）: `name?`, `desc?`, `tags=[]`, `cache=False`、`verify=True` で veriq へ流す

### 5.3 UnifiedRegistry の API

```python
registry.register(kind, definition)
registry.list(kind=None, system=None) -> list[Definition]
registry.get(kind, key) -> Definition | None
registry.json_schema(kind, key) -> dict
registry.openapi_routes() -> list[Route]   # FastAPI が叩く
registry.stubs() -> str                     # gen_stubs が叩く
```

→ **追加機能（新しい consumer）は registry の trait を増やすだけ**で対応。

### 5.4 共通エラーレスポンス

```json
{
  "type": "validation_error",       // or conflict, etag_mismatch, not_found, ...
  "status": 422,
  "title": "Input validation failed",
  "detail": "spec.capacity_wh must be >= 0",
  "errors": [
    { "loc": ["spec", "capacity_wh"], "msg": "...", "type": "...", "input": -10 }
  ],
  "instance": "/api/components/power/battery/main"
}
```

全 endpoint で同型。RFC 7807 互換寄り。

## 6. データの流れ（完成形）

### 6.1 設計者が `Battery` を書く瞬間

```
class Battery(Component, MultiInstance, TemperatureSensitive): ...    # systems/power/components.py
  ↓ __init_subclass__ で auto-register（system はファイルパスから推論）
UnifiedRegistry.components[("power", "battery")] = ComponentDefinition(...)
  ↓ (起動時)
  ├── FastAPI が /api/components/power/battery/* を自動生やし
  ├── CLI が `craft get power battery ...` を自動生やし
  ├── Swagger UI に Battery セクション自動追加
  ├── gen_stubs が BatteryEntry を .pyi に書き出し
  └── doc-generator が markdown に Battery 表を追記
```

### 6.2 ユーザが API でインスタンス追加

```
POST /api/components/power/battery/aux
  ↓ asyncio.Lock(power.toml)
  ↓ Entry.model_validate(body) → 422 on error
  ↓ writer.add_instance(...)  → atomic write
  ↓ audit.log("create", ..., user=...)  → JSONL
  ↓ (opt-in) git commit
  → 201 Created + ETag
```

### 6.3 veriq calc

```
（事前）craft scaffold ... registry → 各 system の data.toml に必須 field を雛形生成
（事前）ユーザが data.toml の値を編集
        ↓
veriq calc / POST /api/verify
  ↓ craft merge ... 全 systems/*/data.toml → generated/merged.toml（自動再生成）
  ↓ load registry + merged.toml
  ↓ build dynamic root model（既存仕組み）
  ↓ vq.evaluate_project(merged.toml) — veriq の依存グラフ走査
  ↓ @analysis(verify=True) も dependency graph に乗っている
  ↓ 結果を generated/runs/{id}/result.toml + DB に保存
  → 結果 ID 返却、GET /runs/{id} で取得
```

scaffold / merge の詳細は [[データパイプライン]]。

### 6.4 LLM 提案ループ ❓

```
入力: 要求仕様（自然言語）
  ↓
LLM agent (tools = JSON Schema from registry)
  ↓ propose: Battery(capacity_wh=120, ...) を追加
  ↓
Entry.model_validate(...) → 失敗なら LLM に feedback
  ↓
TOML に書き込み → veriq calc 実行
  ↓
verification 結果を LLM に feedback
  ↓ 充足するまで反復、もしくは無理と判断したら報告
```

## 7. 統一を決めるポイント（推奨）

| 項目 | 推奨 | 理由 |
|---|---|---|
| field helper | ✅ **`fld()` に統合** | 学習コスト最小、本質は同じ |
| 宣言方式 | ✅ **Component/Config は base class、Analysis のみ decorator** | 文字列引数を type に乗せる（[[コンポーネントデコレータ仕様]] §9）。`@query` は廃止 |
| registry | ✅ **`UnifiedRegistry` に統合** | auto-discovery / stub / doc / API 露出を一度書けば済む |
| `@verification` | ❌ **作らない**、`@analysis(verify=True)` に集約 | decorator 過多回避 |
| API path | ✅ **`/api/...` 直書き** (project prefix なし) | 単一プロジェクト前提を素直に。マルチ化は `/api/v2/projects/{pid}/...` で破壊移行 |
| ETag | ✅ **GET で返す、`If-Match` は任意** | 単一ユーザで競合は稀、過剰防衛回避 |
| schema migration | ✅ **`schema_version` フィールドだけ持つ** | v2 が必要になってから migration コード |
| 自動 git commit | ✅ **opt-in (環境変数 / config)** | 通常は手動、bot 運用時のみ on |
| audit log | ✅ **常時 JSONL** (`core/audit.py`) | 軽量、grep 可、git に依存しない |
| stub 生成 | ✅ **CLI で明示実行**、CI で `--check` | 暗黙生成は副作用が見えにくい |
| auto-discovery | ✅ **`pkgutil` 継続** (副作用は受け入れる) | DX の利点が大きい、テストは fixture で吸収 |

## 8. 拡張ポイント（将来追加するなら）

- **新サブシステム追加**: `systems/<new>/` ディレクトリを置くだけ
- **新 component 追加**: `systems/<sub>/components.py` に `class X(Component, ...):` で書くだけ
- **新 config 追加**: `systems/<sub>/configs.py` に `class X(Config):` で書くだけ
- **新解析追加**: `systems/<sub>/analyses.py` に `@analysis` で書くだけ
- **新 consumer 追加** (例: GraphQL endpoint): registry を走査するだけで動的生成
- **新フィールド型** (例: 単位付きベクトル): `fld()` に型サポート追加 + Pydantic validator
- **横断集計が欲しい**: `@analysis` + `vq.Table` で書く（`@query` は廃止）

→ **「core を触らずに拡張できる」** を維持できる構造になっているはず。

## 9. 明示的にやらないこと（境界線）

- ❌ **抽象化のための抽象化**（veriq を hide する layer など）
- ❌ **3 種以上の decorator** — 必要なら `@analysis(kind=...)` 等のフラグで分岐
- ❌ **multi-tenancy** — 単一ユーザ前提を崩さない
- ❌ **複雑な権限制御** — NoOp フックだけ用意、本気で必要になったら考える
- ❌ **GraphQL 化** — REST + OpenAPI で十分、足りなくなったら専用 endpoint で
- ❌ **マイクロサービス化** — 単一プロセスで完結
- ❌ **専用フォームレンダラの自作** — Web UI 時は既存ライブラリ採用（`@rjsf/core` 等）

| 6 | veriq Python API | ✅ **公開 API あり**（`__all__` に 39 symbol）。subprocess 不要、直接 import |
| 7 | スキーマレス領域 | ✅ **`meta` フィールド方式**（アンダースコアなし） |
| 8 | テスト現実依存 | ✅ **fixture 中心**、E2E は別 TOML（[[テスト戦略]] 参照） |

主要な構造への影響:

### MCP サーバを consumer に追加（§3 レイヤ図への追記）

```
Consumers
  CLI │ Swagger UI │ Web UI (後) │ MCP server │ LLM agent │ Notebook │ CI tools
                                  └──► Claude Code / Desktop が直接叩く
```

### veriq 連携は subprocess 廃止、Python API 直接呼び出しに統一

[[API設計]] §veriq Pass-through API の subprocess 案は廃案。
代わりに **`veriq.evaluate_project()` / `build_traceability_report()` 等を import** して使う（[[veriq仕様メモ]] §公式 Python API）。

### Component に `meta` フィールド自動付与

```python
class Battery(Component, MultiInstance, TemperatureSensitive):
    capacity_wh: float = fld(ge=0, unit="Wh")
    # meta: dict[str, Any] | None = None  ← Component 基底が全派生に自動付与
```

TOML 側:
```toml
[batteries.main]
capacity_wh = 100
[batteries.main.meta]
notes = "選定理由..."
datasheet_url = "https://..."
```

`extra="forbid"` は本体側で維持、`meta` 配下だけが自由。詳細は [[コンポーネントデコレータ仕様]] §meta フィールド。
