# 対処方針 — 懸念事項への推奨アプローチ

> 親: [[Craft]] / 関連: [[懸念事項]] / [[API設計]]

各懸念に対して **推奨案 + 代替案 + トレードオフ** を整理。✅ は採用推奨、🤔 は議論余地あり、❌ は採らない理由を明記。

---

## 基本方針の再設定（重要）

**「リッチな Web UI は後回し。API ファーストで土台を固める」**

Phase 1 (現在〜) の中心は:
- **API 群を網羅的に生やす**（schema 公開 / CRUD / 検証 / 履歴）
- **CLI で全機能を叩けるようにする**（API と機能等価）
- **型・テスト・lint を本気で整備**（pyrefly + ruff + pytest）
- **Swagger UI / OpenAPI で「触れるドキュメント」化** ← これが当面の UI
- 専用 Web UI は「API が安定したら作る」フェーズに後退

→ Web フォーム自動生成・閲覧 UX の懸念（[[懸念事項]] §1.1 / §1.2 / §1.3 / §1.5 / §1.6）は **Phase 2 以降に棚上げ**。
→ 代わりに **§1.2 の「閲覧」は API レスポンス設計の問題** として再定義（→ [[API設計]]）。

## A. 型・スキーマ系（最優先）

### A.1 動的生成モデル × pyrefly  ([[懸念事項]] §2.1)

**✅ 推奨: 段階的な型 stub auto-generate**

1. `@component` の登録時、メモリ上のモデルから `.pyi` を生成する `gen_stubs` CLI を整備
   - 出力先: `experiment/schema/_generated/__init__.pyi`
   - 内容: `BatterySpec`, `BatteryDesign`, `BatteryRequirements`, `BatteryEntry` の型シグネチャ
2. ユーザコードは `from craft.schema._generated import BatteryEntry` で import 可能に
3. CI に `gen_stubs --check` を入れて「stub が最新か」を gate
4. pre-commit hook で同コマンド実行

**🤔 代替: 手書きで Spec/Design クラスを並べる(旧スタイル回帰)**
- 静的解析は完全だが、ボイラープレートが復活 → 本プロジェクトの目的を損なう
- ❌ 採らない

**🤔 代替: registry 経由のみで利用**
- `registry.get_entry_model("power", "battery")` → 戻り値型は `Type[ComponentEntry]`
- 利点: 実装が単純。欠点: IDE 補完が効かず開発体験が悪い
- → API/CLI 内部実装には使うが、**ユーザ向け公開 API は stub を介す**

**トレードオフ**:
- stub の生成・チェックがビルドステップに増える
- 動的 enum の値（インスタンス名）は stub に含められない → §A.2 で別扱い

---

### A.2 動的 StrEnum の型表現  ([[懸念事項]] §2.2)

**✅ 推奨: 「動的 enum は型では守らない」と明文化、ランタイム検証に倒す**

- インスタンス名（`main` / `aux` 等）の typo は **API リクエスト時に 404 / 422** で弾く
- pyrefly では `BatteriesName` は単に `StrEnum` 型として扱う
- 補完支援は **API レスポンス** で代替: `GET /api/instances/power/batteries` でフロント/CLI に列挙提供

**❌ 別案: stub 生成時にインスタンス名も焼き込む**
- TOML が変わるたび stub regen が必要 → Web 追加の利点を消す
- 採らない

**ドキュメント化**: [[コンポーネントデコレータ仕様]] と [[最終構成]] §5 に「動的 enum はランタイム検証のみ」と明記

---

### A.3 registry シングルトンとテスト  ([[懸念事項]] §2.3)

**✅ 推奨: registry をクラスインスタンス化 + テスト用 fixture**

```python
class ComponentRegistry:
    def __init__(self): ...
    def register(self, ...): ...

default_registry = ComponentRegistry()  # 通常はこれを使う

# pytest fixture
@pytest.fixture
def isolated_registry(monkeypatch):
    fresh = ComponentRegistry()
    monkeypatch.setattr("craft.schema._registry.default_registry", fresh)
    return fresh
```

- subsystems の auto-discovery は `default_registry` 前提だが、テストは差し替え可能に
- import 順序問題は「auto-discovery を遅延化（lazy initialization）」で回避

---

### A.4 スキーマ進化 / `extra="forbid"` 運用  ([[懸念事項]] §2.4)

**✅ 推奨: `schema_version` + 専用 migration scripts**

1. TOML に `[meta]` テーブル: `schema_version = 1`
2. `core/migrations/` に `v{N}_to_v{N+1}.py` を置く（pure function: dict → dict）
3. 読み込み時に migration chain を自動適用してから validate
4. CLI: `craft migrate --dry-run` で差分確認

**🤔 代替: 完全な後方互換を諦め、`extra="ignore"` に緩める**
- 一見楽だが typo 検出力が落ちる → ❌ 採らない

---

### A.5 ネスト config / datasheet_only  ([[懸念事項]] §2.5)

**✅ 推奨: `@component` を kind 分岐させる**

```python
@component(kind="datasheet")   # design なし
class SolarCell:
    efficiency: float = cf(...)

@component(kind="config")       # nested 構造を許容
class LinkBudget:
    class BitRates:
        low: float = cf(unit="bps")
        high: float = cf(unit="bps")
    bit_rates: BitRates
```

- `kind` は `"component"` (default) / `"datasheet"` / `"config"` の三値
- 内部実装: kind に応じて生成するモデルセットを変える
- レジストリは `(subsystem, name, kind)` の三項で持つ

**Phase 2 の作業項目に格上げ**。[[課題と検討事項]] から本ノートへ移動済み。

## B. 開発体制系

### B.1 pyrefly セットアップ  ([[懸念事項]] §3.1)

**✅ 推奨: 段階導入 + baseline 方式**

1. `pyproject.toml` に pyrefly 設定追加、初期は **non-strict**
2. 初回フルスキャン結果を `pyrefly-baseline.json` に保存
3. CI は「baseline からエラーが増えていないか」を gate（regression-only）
4. リファクタの一環で徐々に解消、解消したら baseline を更新
5. 新規ファイルは strict 適用

**🤔 trade**: いきなり strict にすると既存コードのエラーで止まる。baseline 方式は安全な漸進改善。

---

### B.2 ruff + `__future__ annotations` 禁止  ([[懸念事項]] §3.2)

**✅ 推奨: ruff 標準 ruleset + 場所限定の禁止ルール**

```toml
# pyproject.toml
[tool.ruff.lint]
select = ["E", "F", "W", "I", "B", "UP", "SIM", "RUF"]

[tool.ruff.lint.per-file-ignores]
# verification/scopes/ では future annotations を絶対に許可しない
# (デフォルトで未使用だが、誰かが追加するのを防ぐ)
```

**禁止の強制**: ruff には直接 `__future__` を禁じるルールがないので、追加で:

```python
# scripts/check_no_future_annotations.py  (pre-commit + CI で実行)
forbidden_dirs = ["experiment/verification/scopes"]
# grep "from __future__ import annotations" し、ヒットしたら exit 1
```

ファイル先頭に **警告コメント** も併用:
```python
# DO NOT add `from __future__ import annotations` — breaks veriq's inspect.signature
```

---

### B.3 テスト戦略  ([[懸念事項]] §3.3)

**✅ 推奨: 「速いテストを厚く、遅いテストを薄く」**

| 層 | 規模 | 何を確かめる | 頻度 |
|---|---|---|---|
| Unit | 厚い | `cf()` / `_strictify()` / `writer` / `dynamic_enum` | 毎回 |
| Contract (snapshot) | 厚い | `@component` 生成物の Pydantic schema | 毎回 |
| Integration | 中 | API ルーター + registry + 実 TOML | 毎回 |
| Property | 中 | TOML round-trip、`add → remove` 不変条件 | 毎回 |
| E2E (CLI) | 薄い | CLI コマンド → TOML 変化 → veriq pass | PR ごと |
| E2E (Web) | なし | Phase 2 で playwright 導入 | - |

**重要**: contract test は **snapshot 形式** で。`@component` の挙動が変わったら snapshot diff で即気付く。

---

### B.4 CI パイプライン  ([[懸念事項]] §3.4)

**✅ 推奨: GitHub Actions で `lint → typecheck → test → schema-gate`**

```
1. ruff check / format check
2. pyrefly (baseline 比較)
3. pytest (unit + contract + integration + property)
4. python -m craft.schema.gen_stubs --check  ← stub 同期チェック
5. python -m craft.schema.validate_all_tomls  ← 全 TOML が現スキーマでロード可能か
6. veriq dry-run (sample design で検証スクリプトが import 失敗しないか)
```

## C. アーキテクチャ系

### C.1 並行書き込み (ロストアップデート)  ([[懸念事項]] §4.1)

**✅ 推奨: 二層防御**

- **L1 (in-process)**: `asyncio.Lock` をファイルパスごとに保持（`dict[Path, asyncio.Lock]`）
  - FastAPI 単一プロセス内のレースを排除
- **L2 (cross-client)**: 楽観的ロック
  - GET レスポンスに `ETag` ヘッダ（TOML の sha256 or mtime）
  - PUT/PATCH/DELETE は `If-Match` 必須、不一致なら 412 Precondition Failed
  - クライアントは 412 を受けたら GET し直して merge or abort

**❌ 採らない**: `fcntl.flock` は multi-process 構成を取らない限り過剰。複雑化のコスト > リターン。

---

### C.2 認証・認可  ([[懸念事項]] §4.2)

**✅ 推奨: Phase 1 は `127.0.0.1` バインドのみ**

- 開発機 / 個人ローカル運用前提。認証は実装しない
- ただし API 設計時点で `Depends(get_current_user)` のフックは入れておく（NoOp 実装）
- 共有環境に出す段階で bearer token / OIDC を後付け

---

### C.3 マルチプロジェクト対応  ([[懸念事項]] §4.3)

**✅ 推奨: `build_project(data_root)` factory 化（Phase 2 で）**

- 現在: `paths.py` の単一参照点
- 移行: `Project(data_root: Path)` クラスに registry / writer / verifier をぶら下げる
- API も `/api/projects/{project_id}/...` の prefix に
- Phase 1 では single-project 前提でも、**API path は将来 `/projects/{id}` を入れられる構造** で設計しておく

---

### C.4 サブシステム間依存  ([[懸念事項]] §4.4)

**🤔 議論継続** — まだ具体ケースが少ない。次の検証実装（thermal）で実例を作ってから方針決定。

- 暫定方針: `vq.Ref` でクロスサブシステム参照を許容、検証順序は **トポロジカルソート** で機械的に決める
- 循環依存は検出時にエラー、人間が「どちらを先に決めるか」を判断

---

### C.5 veriq への依存度  ([[懸念事項]] §4.5)

**✅ 推奨: 抽象化しない、密結合で割り切る**

- veriq に強く依存することで開発スピードを取る
- 抽象層を挟むと（a）開発負荷増 （b）veriq の機能を活かしきれない
- 万一の乗り換えは「その時に書き直す」と腹を括る

---

## D. 運用系

### D.1 バックアップ・履歴  ([[懸念事項]] §5.1 / §5.3)

**✅ 推奨: git に寄せる + アプリ層では薄く**

- `design/data/` 配下を git 管理（既にそうかもしれない）
- API の書き込みごとに自動 commit (`feat(power): add batteries.aux via API`) する常駐プロセス
  - メリット: 履歴・diff・rollback が git の機能で得られる
  - デメリット: 高頻度 commit でログが汚れる → squash 戦略を後で考える
- アプリ層では「最後の N 件の操作ログ」を JSONL で軽く持つ程度

---

### D.2 観測性  ([[懸念事項]] §5.2)

**✅ 推奨: 最小限から始める**

- `logging` 標準 + JSON formatter
- 検証結果は `design/runs/{timestamp}/result.toml` に蓄積（後の分析・LLM 学習用）
- メトリクス収集は Phase 3 以降
