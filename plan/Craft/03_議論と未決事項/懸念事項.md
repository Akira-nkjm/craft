## 1. ユーザビリティ（入力・閲覧）

### 1.1 🔴 JSON Schema → HTML フォーム自動生成の限界
- 素朴な自動生成は **「ただのフォーム」** になりがち。サブシステム設計者の作業ストレスが高い
- 単位（`Wh`, `V`, `W`）の見せ方: ラベル右側に suffix、placeholder、tooltip のどれか
- 数値入力: 指数表記（`1.5e-3`）、有効桁、SI 接頭辞のサポート
- enum: ドロップダウン vs ラジオ vs autocomplete
- `optional + default` の扱い: 「空欄＝デフォルト」を明示するか、初期値を埋めて見せるか
- **検討**: 自前で薄いフォームレンダラを書くか、`json-schema-form` 系ライブラリ（`@rjsf/core` 等）を導入するか

### 1.2 🔴 インスタンス「閲覧」体験
- 現状 CRUD の C/U/D は明確だが **R（一覧・詳細表示）の設計が薄い**
- 必要そうな画面:
  - サブシステム一覧（`power.toml` のテーブル全体を見渡す）
  - インスタンス詳細（spec / design / requirements を並べて表示）
  - サブシステム横断ビュー（衛星全体としての概念設計サマリ）
  - 検証結果ビュー（veriq の出力をどう紐付けて表示するか）

### 1.3 🟡 バリデーション UX
- Pydantic の `ValidationError` をフィールド単位でインライン表示
- サブミット前のリアルタイム検証（onChange で API 叩く vs JSON Schema を JS 側でも評価）
- veriq 検証結果（要求未達など）をフォームに戻して可視化したい

### 1.4 🟡 「現在値との diff」表示
- 既存インスタンスを編集する時、変更前後の差分を見せたい
- 誰が・いつ・何を変えたか（→ 5.3 監査ログと連動）

### 1.5 🟡 フォーム間の依存関係
- 例: `has_power_mode=True` のコンポーネントは `OperationMode` ごとの欄が現れる
- 例: `Battery` の `nominal_voltage_v` を変えると `capacity_wh` の妥当範囲が変わる
- JSON Schema の `dependencies` / `if-then-else` で表現できる範囲には限界がある

### 1.6 🟢 補助機能
- ファイル添付（データシート PDF を spec にリンク）
- インスタンスのテンプレート / クローン
- undo / redo、編集履歴
- 多言語対応（日本語ラベルを `desc` 以外でも持つか）

## 2. 型・スキーマの厳密性

### 2.1 🔴 動的生成モデル × pyrefly の相性
- `@component` で生成される `BatterySpec` / `BatteryEntry` 等は **メタプログラミングなので静的解析できない**
- pyrefly / mypy はクラス本体（`class Battery:`）しか見えず、`BatteryEntry` は import しても unresolved になる可能性
- **対応案**:
  - (a) `@component` を Pydantic の `ModelMetaclass` 派生で実装し、IDE/型チェッカに見えやすくする
  - (b) **型 stub (`.pyi`) を auto-generate** するスクリプトを CI に組み込む
  - (c) 利用側は `registry.get_entry_model("power", "battery")` 経由のみとし、型は `Type[ComponentEntry]` で諦める
- → **方針決定が必要**。pyrefly を真面目に使うなら (b) が現実解か

### 2.2 🔴 動的 StrEnum の型表現
- `load_instance_enum()` で生成される enum は **実行時にしか存在しない**
- veriq scope 内で引数型として使うと、pyrefly からは `StrEnum` としか見えない
- インスタンス名の typo を静的に検出できないトレードオフ
- → これは **「動的追加」と引き換えに諦める** 仕様として明文化が必要

### 2.3 🟡 registry の単一性とテスト
- ComponentRegistry がプロセスグローバルなシングルトン → テスト間で状態リーク
- `pytest` で fresh registry を用意するフィクスチャ設計が必要
- import 順序依存（systems の auto-discovery）

### 2.4 🟡 `extra="forbid"` 運用と TOML 進化
- フィールド追加は基本後方互換だが、**削除・リネームは破壊的**
- マイグレーション戦略:
  - スキーマバージョンを TOML に持たせる（`[meta] schema_version = 2`）
  - 旧→新の変換スクリプトを `core/migrations/` に集約
  - CI で「全 TOML が現スキーマでロードできる」を保証

### 2.5 🟡 ネスト config / datasheet_only の表現
- [[課題と検討事項]] で既出。`@component` API 拡張で対応するか、別デコレータを切るか

### 2.6 🟢 LLM 出力の型保証
- 概念設計を LLM に提案させた時、出力を `Entry.model_validate()` で即拒否できる → 強み
- 逆に LLM への入力に JSON Schema を渡せるので、構造化出力 API（OpenAI structured output、Anthropic tool use）と相性が良い

## 4. アーキテクチャ的懸念

### 4.1 🔴 並行書き込みのロストアップデート
- `writer.add_instance()` は `read → modify → atomic write` のパターン
- **`os.replace` はアトミックだが、read と write の間に他者が書けば消える**（last writer wins）
- 対策案:
  - ファイル単位の `fcntl.flock`
  - 楽観的ロック（ETag / version field を API レスポンスに含めて、PUT 時に検証）
  - 単一ライターキュー（asyncio.Lock）
- Web UI が複数タブ開いた瞬間に問題化する可能性

### 4.2 🔴 認証・認可
- 現状 API は誰でも叩ける前提。社内/個人用途でも最低限のトークン認証は必要か
- 「設計者ロール / レビュアロール / 閲覧のみ」の権限分離が将来必要になりそう
- 当面はローカルバインド（`127.0.0.1`）のみで運用？

### 4.3 🟡 データルートのマルチプロジェクト対応
- 現状 `paths.py` 単一参照点 → 複数衛星プロジェクトの並行管理は不可
- 将来「ミッション A / B」を切り替えたくなる可能性
- `build_project(data_root)` factory 化（[[課題と検討事項]] 既出）が解決策と重なる

### 4.4 🟡 サブシステム間依存
- power ↔ thermal ↔ com の物理的依存（消費電力・熱・通信）
- veriq の `vq.Ref` クロス参照で表現する想定だが、 **検証順序の決定ロジック** が未設計
- どのサブシステムから検証するか、循環依存はどう扱うか

### 4.5 🟡 veriq への依存
- 外部ライブラリへの密結合（`vq.Ref` / `vq.Table` / `vq.Scope` 等）
- veriq の API 変更追従コスト
- 万一の代替検証ツールへの乗り換え可能性 → 抽象層を挟むか、密結合で割り切るか

### 4.6 🟢 フロントの保守性
- Vanilla JS SPA は最初は身軽だが、画面数が増えると保守困難
- 移行の閾値を決めておく（画面 5 つ超えたら React / Vue 検討、など）
