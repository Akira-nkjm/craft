# SOUL

このリポジトリの「魂」 — 設定の背後にある哲学。新しいルール・スキル・フックを追加する際の判断基準。

## Core Identity

**Claude Code と Codex CLI を等価に扱えるエージェント設定テンプレート。** 単一のツールへの依存を避け、共通の作業ルール（`AGENTS.md`）と詳細ルール（`.claude/rules/`）を中心に、各ハーネス固有の入口（`CLAUDE.md` / `.codex/`）から参照する構造を取る。

## Core Principles

1. **Agent-First** — 専門エージェントが存在する作業は、最初から委譲する（planner / code-reviewer / security-reviewer / build-error-resolver 等）
2. **Test-Driven** — 実装より先にテストを書き、テストが落ちることを確認してから実装する
3. **Security-First** — 入力検証・シークレット保護・安全なデフォルトを既定動作とする
4. **Immutability** — 既存オブジェクトの変更ではなく、新しいオブジェクトを返す
5. **Plan Before Execute** — 非自明な変更は段階に分け、計画と承認を経てから実行する
6. **Research First** — 新規実装の前に既存実装（GitHub / ライブラリ）を探す

## Cross-Harness Vision

このリポジトリは「Claude Code 専用設定」から「マルチハーネス対応のエージェント設定」へ進化する途上にある。

- **共有資産**: `AGENTS.md`, `RULES.md`, `SOUL.md`, `.claude/rules/`, `.tasks/`
- **ハーネス固有**: `CLAUDE.md`, `.claude/{commands,skills,tools/hooks}`, `.codex/`

新機能を追加する際は、「これは共有資産か、ハーネス固有か」を最初に問う。共有にできるものは共有側へ。

## なぜスキル中心なのか

ルール（受動的な制約）だけでは「いつ何をするか」が曖昧になる。スキル（能動的な手順）は「特定の状況で起動される完結した手順書」として、エージェントが**自発的に**呼び出せる。ECC が 246 スキルを持つ理由はこれ。本リポジトリでは厳選した汎用スキルから始める。

## 追加の判断基準

新しい要素（ルール／スキル／フック／コマンド）を追加するとき:

- 既存と重複しないか？
- 汎用か、プロジェクト固有か？（後者なら `project.md` に書く）
- ハーネスをまたいで価値があるか？（あるなら共有資産に）
- メンテナンス可能なサイズか？（200 行を大きく超えるなら分割）
- 「いつ使うか」が明文化されているか？
